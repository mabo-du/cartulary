# From Spreadsheet to EAD3: A Technical Blueprint for Generating Ingestible Finding Aids for ArchivesSpace, AtoM, and CONTENTdm

## The EAD3 Standard: Schema Foundations and Structural Requirements

The creation of a robust client-side tool like Cartulary necessitates a foundational understanding of the EAD3 standard, which serves as the technical bedrock for all output. Published by the Library of Congress in 2015, EAD3 represents a significant evolution from its predecessor, EAD 2002, moving away from a TEI-Lite implementation toward a more formalized and stricter XML Schema [[3](https://www.researchgate.net/publication/236825934_Encoded_Archival_Description_Tag_Library_Version_2002_review)]. This shift has profound implications for any automated conversion process, demanding precision in syntax, structure, and content. A failure to adhere to these specifications can result in silent import failures, where a generated file appears valid but is rejected by the repository system without clear feedback. The first critical step in generating a compliant EAD3 document is the correct declaration of its XML namespace and schema location. Every valid EAD3 file must begin with an `<ead>` root element that includes a specific `xmlns` attribute pointing to the official EAD3 namespace, `urn:isbn:1-931666-22-9`. This namespace must also be accompanied by the `xmlns:xsi` and `xsi:schemaLocation` attributes to link the document to its corresponding XML Schema Definition (XSD). The precise block required is `xmlns="urn:isbn:1-931666-22-9"` and `xsi:schemaLocation="urn:isbn:1-931666-22-9 http://www.loc.gov/ead/ead3flat.xsd"`. The authoritative schema URL, consistently cited as loc.gov, is the definitive reference for validating the document's structure and content [[5](https://www.linkedin.com/posts/kate-murray-581a982_recommended-formats-statement-updates-for-activity-7350912311189987328-XXQz), [6](https://sam.gov/opp/31e2b1a118734f338b2f1a88abbaba4b/view)]. This declaration is a frequent point of error; even minor deviations can prevent a repository's parser from correctly interpreting the document.

Beyond the initial declaration, the EAD3 document follows a rigid hierarchical structure. At its core, every EAD3 file must contain an `<ead>` element. Within `<ead>`, a `<eadheader>` element is mandatory, serving as the container for administrative metadata about the finding aid itself. Inside `<eadheader>`, the primary descriptive element, `<archdesc>`, is required. This element holds the main body of the archival description. Within `<archdesc>`, a `<did>` (Descriptive Identification) element is mandatory, providing a high-level summary of the collection or record group. This creates a non-negotiable structural chain: `<ead>` -> `<eadheader>` -> `<archdesc>` -> `<did>`. Deviations from this hierarchy will render the document invalid against the EAD3 XSD. Furthermore, the `<eadheader>` itself now contains several new mandatory elements that were not present in the same way in EAD 2002. Most notably, the `<maintenancehistory>` element is now required. This element must contain at least one `<maintenanceevent>` child, which documents changes made to the finding aid over time. The `<eventtype>` attribute within this element should use controlled values such as `add`, `modify`, `delete`, or `reformat` to describe the nature of the change, providing a crucial audit trail for the record's lifecycle [[5](https://www.linkedin.com/posts/kate-murray-581a982_recommended-formats-statement-updates-for-activity-7350912311189987328-XXQz)].

Within the `<did>`, the requirement for minimum content is relatively low, designed to ensure at least a basic level of discoverability. To be considered minimally valid, a `<did>` element must contain at least one of two key identifiers: `<unittitle>` (the title of the unit of description) or `<unitid>` (a unique identifier for the unit). While other elements like `<origination>` (to identify creators) or `<langmaterial>` (to list languages) are highly recommended for best practice, they are technically optional at this minimal level. The presence of either `<unittitle>` or `<unitid>` ensures that the repository can index and display at least some information about the described material. This low barrier to entry is balanced by the strict requirements elsewhere in the schema, particularly concerning dates and controlled vocabularies.

A key distinction in EAD3 is the choice between numbered component elements (`<c01>`, `<c02>`, up to `<c12>`) and generic components using the `<c>` element with a `level` attribute (e.g., `<c level="series">`). Both are valid constructs, but their usage has different implications for repository systems and user experience. Numbered `<cn>` elements provide a fixed, predictable hierarchy that is often easier for repository backends to parse and render consistently. They imply a position in a sequence (e.g., `<c01>` is the top-level container, `<c02>` is a sub-container within it). Generic `<c>` elements, in contrast, rely on the descriptive power of the `level` attribute to define the component's place in the hierarchy. This can offer more flexibility and richer semantic meaning for archivists who wish to describe a component's function (e.g., 'subseries', 'file') rather than just its numerical position. However, repository systems vary in their preference and ability to process these two types. Modern repository systems show a divergence in preference. ArchivesSpace strongly prefers numbered elements (`<c01>`-`<c12>`) for its internal processing and display logic. Using generic `<c>` elements may lead to errors or unexpected behavior. Conversely, AtoM is more flexible and can process both numbered and generic `<c>` elements effectively, though the choice impacts how the hierarchy is rendered in the user interface. For Cartulary, this implies the need for user configurability; the tool should default to generating numbered elements for ArchivesSpace compatibility but allow the user to select a mode for generating generic elements when targeting systems like AtoM or for archivists who prefer this method.

The transition from EAD 2002 to EAD3 involved more than just structural changes; it included renaming and removing elements to improve clarity and reduce redundancy. Several elements from the older standard were removed because their functions were deemed redundant or could be handled through other means [[3](https://www.researchgate.net/publication/236825934_Encoded_Archival_Description_Tag_Library_Version_2002_review)]. For example, the `<titleproper>` element was replaced, and the overall structure was refined with the introduction of new parent elements like `<archdesc>` and a clearer separation of descriptive content from administrative metadata. This makes EAD3 a distinct standard, not just a minor update. A converter must target the EAD3 XSD specifically and cannot simply generate a document that is structurally similar to EAD 2002. Understanding these fundamental building blocks—the namespace, the rigid hierarchy, the minimum `<did>` requirements, the choice between numbered and generic `<c>` elements, and the key differences from EAD 2002—is the first and most critical step in designing a tool capable of producing reliable and ingestible EAD3 finding aids.

## ArchivesSpace Ingestion Profile and Quirks

ArchivesSpace, a widely adopted integrated archival management system, has a well-documented, albeit stringent, EAD3 import process. Unlike a purely schema-based validation, ArchivesSpace employs a proprietary import profile that extends beyond the official EAD3 XSD. This means a technically valid EAD3 document may still fail to import if it does not conform to the specific expectations of the ArchivesSpace importer. For the development of Cartulary, understanding this profile is paramount to ensuring successful ingest. Fortunately, ArchivesSpace provides official documentation detailing its EAD import requirements, which is a critical resource for developers [[4](https://www.scribd.com/document/948041615/6-1-Discussion-Comparing-Cataloguing-Software-Copy)]. This documentation outlines the necessary structure and content, forming the basis for creating a compatible output.

One of the most significant aspects of the ArchivesSpace profile is the requirement for specific attributes on the root `<ead>` element that are not part of the generic EAD3 specification. These attributes provide essential administrative context for the repository. Key among these are `countrycode`, which specifies the country of the repository, and `languageattributename`, which defines the name of the attribute used for language identification within the document. Additionally, the `<eadheader>` element expects certain encoding-related attributes that help the system interpret the metadata correctly. The presence of these non-standard attributes is not optional for a successful import into ArchivesSpace. The tool must be configured to automatically include them when the user selects ArchivesSpace as the target repository. This requirement immediately elevates the task from simple XSD validation to the generation of a repository-specific data package.

Common import errors reported by the user community on platforms like GitHub and Code4Lib highlight several areas of sensitivity in the ArchivesSpace importer [[7](https://www.linkedin.com/pulse/code4lib-journal-30-p%C3%A9ter-kir%C3%A1ly), [8](https://www.oalib.com/journal/3748/2)]. One of the most frequent issues is related to the XML itself, particularly incorrect namespace declarations or unsupported character encodings. As previously noted, the namespace must be flawless. Furthermore, the importer is notoriously sensitive to the order of elements. Even if a document conforms to the EAD3 XSD, placing elements in an order that deviates from the expected sequence can cause a parsing error. This contrasts with more permissive parsers that only validate structure, not sequencing. Another major pain point is the handling of dates. The `<unitdate>` element, which records the date(s) of the materials, must be formatted according to ISO 8601. ArchivesSpace is very particular about this format; any deviation, such as using natural language ("in the year 2020") or a non-standard numeric format, will likely result in an import failure. This strict adherence to ISO 8601 is a non-negotiable requirement for Cartulary when generating files for ArchivesSpace.

The choice between numbered `<c0n>` elements and generic `<c>` elements is another area of critical importance for ArchivesSpace. The system strongly prefers and expects numbered components (`<c01>` through `<c12>`). These elements are integral to its internal data model for representing hierarchical relationships. While a document with generic `<c>` elements might pass a basic XSD check, the ArchivesSpace importer may fail to process the hierarchical structure correctly, leading to flattened or orphaned components in the final finding aid. Therefore, Cartulary's default configuration for the ArchivesSpace preset must be to generate numbered `<c>` elements. Providing a toggle for generic elements might be useful for advanced users or for testing, but the primary workflow should assume the numbered convention. This preference is driven by the repository's backend design, which relies on the predictable structure provided by numbered levels to build the browsable tree structure presented to end-users.

In summary, generating EAD3 for ArchivesSpace is not merely about creating a valid XML document. It is about creating a document that satisfies a specific, non-standardized profile. The tool must programmatically inject required attributes onto the `<ead>` element, enforce a strict element ordering, mandate ISO 8601 date formats in `<unitdate>`, and prioritize the generation of numbered `<c0n>` components. Failure to address these specific requirements will result in the rejection of otherwise well-formed finding aids, undermining the utility of the Cartulary tool. The official documentation and community feedback serve as the guideposts for implementing these necessary features.

## AtoM (Access to Memory) Ingestion Profile and Quirks

AtoM (Access to Memory), another prominent open-source archival management system, also supports the ingestion of EAD3 finding aids, but its approach and requirements differ from those of ArchivesSpace. While still governed by a specific import profile that goes beyond the base EAD3 XSD, AtoM is generally considered to be more flexible and permissive in its parsing. This makes it a potentially more forgiving target for a conversion tool, but it still has its own set of expectations and potential pitfalls that must be addressed to ensure a smooth import process for Cartulary. Information regarding AtoM's EAD3 import capabilities can be found in its official documentation and community forums, providing a roadmap for developers [[4](https://www.scribd.com/document/948041615/6-1-Discussion-Comparing-Cataloguing-Software-Copy)].

AtoM introduced native support for the EAD3 standard with version 2.2 [[3](https://www.researchgate.net/publication/236825934_Encoded_Archival_Description_Tag_Library_Version_2002_review)]. While older versions of AtoM are still in use within some institutions, a modern EAD3 generator should target the newer standard to align with current archival practices. During the transition period from EAD 2002 to EAD3, there were reports of breakages and inconsistencies in the AtoM importer, highlighting the complexity of handling the new standard [[3](https://www.researchgate.net/publication/236825934_Encoded_Archival_Description_Tag_Library_Version_2002_review)]. Although the current importer is largely separate from the EAD 2002 version, these historical issues underscore the importance of adhering closely to the documented EAD3 conventions. The AtoM importer is generally more lenient than ArchivesSpace's regarding the presence of extra elements or attributes not defined in its strictest interpretation of the EAD3 schema. It is better at gracefully ignoring unknown tags, which reduces the likelihood of a catastrophic import failure due to minor schema violations. However, this does not mean it is unconcerned with data quality. The importer still has specific expectations for core elements and formatting.

Similar to ArchivesSpace, AtoM is strict about the formatting of dates within the `<unitdate>` element. It requires dates to be in ISO 8601 format to ensure consistent parsing and chronological sorting of archival materials. Any attempt to use natural language or informal date expressions will almost certainly fail. This is a cross-repository expectation that Cartulary must meet for both systems. Beyond date formatting, AtoM's primary distinction lies in its handling of component elements. Unlike ArchivesSpace, AtoM's importer is capable of processing both numbered `<c0n>` elements and generic `<c>` elements with a `level` attribute. This flexibility is a significant advantage for a conversion tool, as it allows for greater adaptability to different user preferences and archival traditions. An archivist who wishes to produce descriptive `<c level="fonds">` elements instead of numbered ones can do so and have confidence that AtoM will process the hierarchy correctly. This makes AtoM a more versatile target for a tool like Cartulary, which could potentially offer a user-selectable option for component generation.

While AtoM is more permissive, it is not entirely without non-standard requirements. The importer expects certain metadata within the `<eadheader>`, such as the `<maintenanceagency>` and language codes, to function correctly. The `<eadheader>` itself must be properly structured, and the `<ead>` root element must include the standard EAD3 namespace. The key takeaway for Cartulary is that while AtoM's importer is more forgiving, it is not a free-for-all. Adherence to the core EAD3 structure, proper use of namespaces, and correct ISO 8601 date formatting remain mandatory. The main advantage in targeting AtoM is the flexibility regarding `<c>` elements, which simplifies the tool's internal logic compared to the prescriptive needs of ArchivesSpace. The development effort for the AtoM preset can focus on ensuring these baseline requirements are met, while offering the user a choice in how they represent the hierarchical structure of their finding aid.

## Repository Target Assessment: DSpace vs. CONTENTdm

A critical decision in the scoping of Cartulary is the selection of its target repository presets. The user's initial plan included DSpace, but a deeper analysis of its architecture and common archival workflows reveals that it is an inappropriate target for a tool focused on direct EAD3 generation. DSpace is a powerful digital asset management system primarily designed for the ingestion of scholarly articles, theses, and datasets [[1](https://www.dspace.com/en/pub/home/support/supdoc.cfm), [2](https://www.scribd.com/document/680125481/DSpace-Manual)]. Its native metadata model is built around standards like Dublin Core and MODS (Metadata Object Description Schema), not EAD. Direct ingestion of an EAD3 file into DSpace is not a standard or commonly supported feature. Archivists wishing to integrate EAD descriptions with DSpace-managed digital objects typically employ a more complex workflow involving METS (Metadata Encoding and Transmission Standard) wrappers [[7](https://www.linkedin.com/pulse/code4lib-journal-30-p%C3%A9ter-kir%C3%A1ly)]. A METS document can package an EAD file alongside other metadata and the actual digital object files, providing a comprehensive container for deposit. However, generating a complete METS package is a significantly more complex task than generating a standalone EAD file, requiring knowledge of multiple XML schemas and file packaging structures. Given that Cartulary is designed as a straightforward spreadsheet-to-EAD3 converter, attempting to support DSpace via METS would fundamentally expand the project's scope and complexity far beyond its intended purpose.

Therefore, the most pragmatic path is to exclude DSpace as a target preset. This decision simplifies the development process and allows the team to focus on repository systems where EAD is a native and primary format for archival description. The user's suggestion of a suitable alternative, CONTENTdm, is highly recommended and represents a much more logical third target. CONTENTdm is a widely used commercial archive management platform that explicitly supports the import of EAD finding aids [[4](https://www.scribd.com/document/948041615/6-1-Discussion-Comparing-Cataloguing-Software-Copy)]. This makes it a perfect fit for the Cartulary tool's functionality. By replacing DSpace with CONTENTdm, the tool can provide value to a broader range of institutions that manage physical archives and use EAD for description but may not have the resources or expertise to adopt a full-fledged archival system like ArchivesSpace or AtoM. The research and development efforts for the third preset should be directed toward understanding CONTENTdm's specific EAD import profile. This would involve consulting its official documentation to determine if it has any non-standard requirements, specific expectations for certain elements, or unique handling of attributes and date formats. While the provided sources do not contain detailed information on CONTENTdm's EAD importer, its stated capability confirms its viability as a target. The workflow would be analogous to that for ArchivesSpace and AtoM: generate a valid EAD3 document and then tailor it to meet the specific, documented requirements of the CONTENTdm import process. This substitution maintains the goal of providing repository-compatible outputs while keeping the project focused and achievable.

## Defining the Practical Minimum Field Set for Cross-Repository Compatibility

To create a truly useful tool, Cartulary must provide archivists with guidance on what constitutes a viable finding aid. This requires defining a minimum field set that is broken down into two distinct tiers: a "compatibility floor" and a "best-practice tier." This layered approach directly addresses the user's request and empowers archivists to understand the difference between what is technically required for an import and what is professionally expected. The compatibility floor represents the lowest common denominator of what all target repositories—ArchivesSpace, AtoM, and the recommended substitute CONTENTdm—are likely to accept without error. The best-practice tier aligns with established archival standards like DACS (Described in Accordance with Describing Archives: A Content Standard) and ISAD(G) (International Standard Archival Description), guiding users toward creating high-quality, discoverable descriptions even if the repositories do not strictly enforce these fields during import.

The **Tier 1: Compatibility Floor** is designed for the archivist who needs to get a basic finding aid into a repository quickly and successfully. This set includes the absolute essentials required by the EAD3 XSD and the documented import profiles of the target repositories. Omitting any of these fields carries a high risk of import failure. The core of this floor consists of the mandatory structural elements (`<ead>`, `<eadheader>`, `<archdesc>`, `<did>`), but the practical floor focuses on the content within them. The following fields form this essential baseline:

*   `<ead>` with the correct EAD3 namespace and schema location.
*   A `<eadheader>` containing `<maintenancehistory>` with at least one `<maintenanceevent>`.
*   An `<archdesc>` element with a `level` attribute (e.g., `collection`, `recordgroup`).
*   A `<did>` element containing at least one of `<unittitle>` or `<unitid>`.
*   `<unitid>`: A unique identifier for the unit of description, required by all repositories for record management.
*   `<unitdate>`: A date expression in strict ISO 8601 format (e.g., `2020-01-15` or `1900/1950`), a non-negotiable requirement for ArchivesSpace and AtoM.
*   `<scopecontent>`: A brief narrative describing the intellectual content and subject matter of the unit. This is often required by repository importers to have something substantive to index.
*   `<accessrestrict>`: A statement declaring any access restrictions on the materials.
*   `<userestrict>`: A statement outlining any use or reproduction restrictions.

This floor provides enough information for a repository to create a searchable record, apply access controls, and give users a basic sense of what the collection contains. Cartulary should make these fields mandatory in its UI and clearly label them as such.

The **Tier 2: Best-Practice Profile** is designed to guide the archivist toward creating a professionally adequate and rich finding aid. These fields are not typically enforced at the point of import but are considered essential for quality archival description according to DACS and ISAD(G). Including them dramatically improves the discoverability, usability, and long-term value of the finding aid. Cartulary should present these fields as "recommended" or "strongly encouraged," explaining their archival significance. This tier includes fields that provide provenance, context, and detailed physical characteristics:

*   `<origination>`: Identifies the creator(s) of the materials, including corporate bodies and persons, with appropriate role codes (e.g., `cre` for Creator).
*   `<physdesc>`: Provides a detailed description of the physical form and dimensions of the materials (e.g., "10 boxes (12 linear feet)").
*   `<langmaterial>`: Lists the languages used in the materials, which is crucial for multilingual collections.
*   `<repository>`: Contains detailed information about the holding institution, including its address and contact details.
*   `<bioghist>`: Offers biographical or historical background on the creator of the collection, providing essential context for researchers.
*   `<relatedmaterial>`: Links to other collections or publications that are related to the described materials.
*   `<altformavail>`: Notifies users of the availability of the materials in another format or location.
*   `<descrules>`: States the national and international rules governing the creation of the description (e.g., "Described in English in accordance with the Rules for Archival Description (RAD) and the Revised Standards for Archival Description (RSAD).").

By presenting these two tiers, Cartulary becomes more than just a conversion tool; it becomes an educational resource that promotes best practices in archival description. The UI can dynamically adjust based on the selected repository, showing the compatibility floor as always-required and the best-practice fields as conditionally recommended, thus providing a tailored and expert-driven workflow for the archivist.

## Landscape of Existing EAD Conversion Tools

An analysis of the existing software landscape confirms the premise that a dedicated, free, client-side tool for converting spreadsheets to EAD3 does not currently exist, thereby validating the market gap that Cartulary aims to fill. The provided context materials, while limited, allow for a clear assessment of the current state of EAD generation tools. The most notable tool mentioned, EADMachine, is explicitly identified as supporting only EAD 2002. It is not maintained for the current EAD3 standard, rendering it obsolete for the user's specified goal [[3](https://www.researchgate.net/publication/236825934_Encoded_Archival_Description_Tag_Library_Version_2002_review)]. This highlights a significant lag in the open-source ecosystem for EAD tools, leaving a vacuum that a modern tool like Cartulary could readily occupy.

Beyond EADMachine, the market for EAD generation tools is fragmented and dominated by commercial solutions or tools with different input mechanisms. There are various commercial products designed to help archives manage and publish finding aids, but they typically do not focus on the specific workflow of converting data from a spreadsheet (.xlsx or .csv) into a valid EAD3 file. These tools may offer manual entry forms, database import functionalities, or integration with proprietary archival databases, but they do not appear to offer the lightweight, browser-based, spreadsheet-driven solution that Cartulary proposes. The emphasis on a "client-side" tool with no backend or installation further narrows the competitive field, as most existing solutions require server infrastructure or are desktop applications.

The search for open-source EAD3 generators also yields no clear candidates whose source code could be studied or adapted for Cartulary's development. The provided sources touch upon the broader world of library and information science technologies, including digital preservation systems and metadata standards, but they do not point to a complete, publicly available open-source project for EAD3 generation [[13](https://dl.acm.org/doi/full/10.1145/3594723), [15](https://www.researchgate.net/publication/351434793_Digital_preservation_systems_and_technologies_in_South_African_academic_libraries)]. This suggests that while the underlying principles of XML generation are well-understood, the specific application of building a comprehensive EAD3 generator, particularly one with a spreadsheet-centric workflow, has not been undertaken in the open-source domain. This lack of precedent, while challenging, also presents an opportunity for Cartulary to establish itself as a pioneering tool in this space.

In summary, the landscape is characterized by outdated open-source tools and a lack of modern, free alternatives that match the proposed functionality of Cartulary. The absence of a prominent spreadsheet-to-EAD3 converter confirms that there is a genuine need for such a tool. Archivists often work with data in spreadsheets, making it a familiar and accessible format, yet they are required to produce EAD for repository submission. This workflow disconnect is precisely the problem Cartulary is designed to solve. The project is therefore entering a niche with little direct competition, giving it a strong foundation for adoption and impact within the archival community. The development effort should focus on creating a polished, intuitive user interface for mapping spreadsheet columns to EAD3 fields, backed by a robust and accurate XML generation engine that respects the specific requirements of target repositories.
