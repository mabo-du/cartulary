# Reverse-Engineering the CONTENTdm EAD Import Profile: A Guide for Repository Developers

## EAD Standard and Namespace Conventions

The foundational question for establishing an EAD import profile for OCLC's CONTENTdm system is determining the precise version of the Encoded Archival Description standard it supports. This inquiry extends to the specific XML namespace declarations and schema requirements that govern the structure of a compliant finding aid. The absence of official documentation from OCLC has necessitated a reliance on secondary sources, primarily institutional examples of EADs hosted within CONTENTdm repositories [[14](https://findingaids.library.columbia.edu/ead/cul-10491409.xml)]. These real-world implementations provide the most direct evidence of the system's actual parsing capabilities, forming the bedrock of this research profile. The analysis reveals that CONTENTdm's importer is fundamentally aligned with the mature and widely adopted EAD 2002 standard, as defined by its associated namespace. While the newer EAD3 standard exists, there is no conclusive evidence within the provided sources to confirm its support.

The definitive identifier for an EAD document is its root element, `<ead>`, which includes a namespace declaration. The single most critical piece of evidence comes from an EAD file for the "Mariana Griswold Van Rensselaer Papers" hosted by Columbia University Libraries, accessible via a CONTENTdm repository [[14](https://findingaids.library.columbia.edu/ead/cul-10491409.xml)]. The root element of this file explicitly declares the namespace `http://www.loc.gov/ead/`. This namespace is the official and exclusive identifier for the EAD 2002 standard, a specification that has been the archival community's principal reference guide for nearly two decades [[1](https://www.researchgate.net/publication/236825934_Encoded_Archival_Description_Tag_Library_Version_2002_review), [15](https://www.academia.edu/22805286/BlogForever_D3_1_Preservation_Strategy_Report)]. The presence of this specific declaration in a finding aid that is successfully rendered and utilized within a live CONTENTdm environment provides strong circumstantial evidence that the system's parser is designed to recognize and process documents conforming to the EAD 2002 schema. Given the stability and long-standing adoption of EAD 2002, it is logical to conclude that CONTENTdm's import mechanism prioritizes compatibility with this established standard. The lack of discourse around EAD3 or its corresponding namespace (`http://ead3.archivists.org/schema/`) in the available community discussions further suggests that support for the newer standard may not be a primary feature or may be less robustly tested [[16](https://www.academia.edu/82306246/Information_and_Knowledge_Organisation_in_Digital_Humanities)].

While the namespace declaration points strongly toward EAD 2002, the requirements for additional schema information, such as the `xsi:schemaLocation` attribute, remain ambiguous. The `xsi:schemaLocation` attribute is a common practice in XML to associate a namespace with its corresponding Document Type Definition (DTD) or XML Schema (XSD), allowing validators to perform more rigorous checks against a formal definition. However, the provided institutional EAD example from Columbia University does not include this attribute [[14](https://findingaids.library.columbia.edu/ead/cul-10491409.xml)]. This absence could imply several possibilities: the CONTENTdm parser might rely solely on internal rules rather than external schemas; it might validate against an internal DTD or XSD without requiring the `schemaLocation` to be declared in the document itself; or the inclusion of the attribute might be optional. The provided context materials contain no forum posts, articles, or technical notes specifically mentioning a requirement for a particular schema location URL. In a system where such a requirement would likely cause widespread import failures among institutions still using EAD 2002, one would reasonably expect numerous community reports detailing such an issue. The silence on this matter across the available Code4Lib archives, GitHub discussions, and user group materials suggests that requiring a specific `xsi:schemaLocation` is not a common failure point and may not be a strict prerequisite for import success. Therefore, while including it for best practices is advisable, it cannot be confirmed as a mandatory requirement based on the available evidence.

Another key aspect of the EAD standard is the use of the `@audience` attribute on the `<ead>` element, which specifies whether the finding aid is intended for internal or external audiences. This attribute can influence how certain elements, like access restrictions, are displayed. The provided source materials, encompassing institutional examples, community discussions, and general documentation snippets, contain no mention of a required `@audience` attribute for CONTENTdm. The EAD from Columbia University, which functions correctly within the system, also omits this attribute entirely [[14](https://findingaids.library.columbia.edu/ead/cul-10491409.xml)]. This lack of information makes it impossible to definitively state whether CONTENTdm enforces its use. It is plausible that the system defaults to a specific audience value (likely "external") if the attribute is not present, a common behavior in many XML-processing systems. Without explicit confirmation from official sources or clear reports from the community about errors related to a missing `@audience` attribute, it must be considered an optional field. Consequently, a repository preset aiming for broad compatibility should not enforce its presence.

Finally, the investigation into non-standard or proprietary attributes added to the `<ead>` element or other parts of the document yields no positive findings. The goal was to identify any unique attributes, similar to those found in other systems like ArchivesSpace (e.g., `countrycode`), that CONTENTdm might require for proper processing. The institutional EAD example from Columbia University is clean and adheres strictly to the EAD 2002 tag library, containing no extraneous or system-specific attributes [[14](https://findingaids.library.columbia.edu/ead/cul-10491409.xml)]. The absence of any such attributes in a working example, combined with the lack of community discussion about them, indicates that CONTENTdm's importer is designed to work with standard-compliant EAD 2002 documents. It does not appear to impose its own extensions on top of the existing schema. This adherence to standards simplifies the task of preparing EAD files for import, as developers do not need to add or modify attributes beyond what is specified in the official EAD 2002 documentation [[1](https://www.researchgate.net/publication/236825934_Encoded_Archival_Description_Tag_Library_Version_2002_review)]. This conclusion aligns with the expectation that a mature, enterprise-level system like CONTENTdm would prioritize interoperability through standard compliance rather than introducing proprietary modifications that could hinder data exchange.

## Minimum Required Fields and Structural Hierarchy

Determining the minimum viable set of fields and the structural hierarchy required for a successful EAD import into CONTENTdm is crucial for developing a reliable and efficient workflow for creating finding aids. This analysis moves beyond the high-level standard to the practical, granular details of what an importing system will absolutely need to create a basic record. By examining the structure of a functional EAD file from a live CONTENTdm repository, alongside logical inferences drawn from archival description principles, a clear picture of the system's baseline expectations emerges. The evidence suggests that CONTENTdm requires a complete, albeit minimal, descriptive hierarchy starting from the `<ead>` root down to essential container elements and their constituent parts. While the system appears flexible in some areas, such as component numbering, it seems to demand a well-formed document with core metadata elements present.

The most instructive source for this analysis is again the EAD file for the Mariana Griswold Van Rensselaer Papers, which is successfully integrated into a Columbia University Libraries' CONTENTdm collection [[14](https://findingaids.library.columbia.edu/ead/cul-10491409.xml)]. This file serves as a practical blueprint for a minimally sufficient finding aid. Its structure begins with the `<ead>` root element, followed by the `<eadheader>`, which contains administrative metadata about the finding aid itself. Immediately following is the `<archdesc>` element, which is the heart of the archival description. Critically, this element is not empty; it possesses a `@level` attribute, in this case, `collection`. The `@level` attribute is fundamental as it defines the scope and nature of the described materials, and its presence is a strong indicator that it is a required part of the `<archdesc>` element. Without a defined level, the system would lack the most basic categorization for the collection.

Within the `<archdesc>`, the next indispensable element is `<did>`, which stands for "Descriptive Identifier." This element acts as a container for the primary title and identifier of the entire finding aid. Inside the `<did>`, the EAD from Columbia includes three key components: `<unittitle>`, `<unitid>`, and `<unitdate>`. The `<unittitle>` provides the main title of the collection, while the `<unitid>` offers a unique identifier, often a catalog number or accession number, which is vital for linking the finding aid to the digital objects it describes. The `<unitdate>` provides date information, which is essential for chronological sorting and searching. The consistent presence of these three elements within the `<did>` in a working example strongly implies that a combination of a title and an identifier is mandatory. While the EAD 2002 standard allows for variations, the practical implementation in CONTENTdm appears to demand at least one of each type. It is possible that a finding aid could exist with only a `<unittitle>` and `<unitid>`, omitting `<unitdate>`, but this remains an open question due to the lack of testing data. Similarly, the presence of `<dsc>` (Description Scope and Content) is standard, but it is often left empty in simple finding aids. Whether an empty `<dsc>` is acceptable or if at least one `<c>` element is required is another area where definitive confirmation is lacking.

The overall structural hierarchy expected by CONTENTdm can be reconstructed from the institutional example. It follows the canonical EAD 2002 structure but highlights the mandatory nodes within it:
`<ead>`
└── `<eadheader>`
└── `<archdesc>` (with `@level` attribute)
    └── `<did>`
        ├── `<unittitle>`
        ├── `<unitid>`
        └── `<unitdate>` (presence inferred but not confirmed as strictly mandatory)

This hierarchical requirement ensures that the system can build a coherent record. It starts with the highest level of metadata (`<eadheader>`), moves to the scope of the description (`<archdesc>`), identifies the collection (`<did>`), and populates its essential characteristics (`<unittitle>`, `<unitid>`, `<unitdate>`). Any deviation that removes a critical branch from this tree, such as omitting the `<did>` or the `<archdesc>`, would almost certainly lead to an import failure or a malformed record. For instance, an error message like "import content failed" has been reported in various contexts when required data is missing or improperly formatted [[4](https://community.adobe.com/questions-611/import-content-failed-error-412228), [20](https://www.dell.com/support/kbdoc/en-us/000234595/dell-vxrail-failed-uploading-meta-bundle-when-creating-advisory-report)]. While these specific instances were not with EAD, they illustrate the type of generic failure that can occur when a system's structural expectations are not met.

Furthermore, the concept of a "minimum field set" extends to the very lowest level of the finding aid, the `<c>` (or component) element. The research question posed whether CONTENTdm could import a finding aid with only `<c>` components and no `<dsc>`. While an empty `<dsc>` is structurally sound, a finding aid with no `<dsc>` at all would violate the EAD standard's expected structure. The `<dsc>` is the designated container for all components. However, the more practical question is whether a finding aid needs multiple `<c>` elements or if a single one is sufficient. The provided materials do not contain an example of a one-component finding aid, so this is unconfirmed. However, given that the purpose of a finding aid is to describe a collection's structure, it is logical to assume that at least one `<c>` element is required to represent the top-level container of the collection's contents. The institutional example from Columbia includes numerous `<c>` elements, but the minimum number needed for a successful, non-empty import is not specified [[14](https://findingaids.library.columbia.edu/ead/cul-10491409.xml)]. This remains an open question that would require empirical testing.

In essence, CONTENTdm's importer expects a well-formed EAD 2002 document that adheres to a specific, non-negotiable path of essential elements. The `<ead>`, `<eadheader>`, `<archdesc>`, and `<did>` are all structural pillars. Within `<did>`, a `<unittitle>` and a `<unitid>` appear to be mandatory for creating a usable record. The `<archdesc>` must declare its level of description. While some fields like `<unitdate>` and the exact number of `<c>` elements may offer some flexibility, the core structure must be present. This understanding allows developers and archivists to focus on reliably generating these key components, confident that as long as the structure is sound, the system will be able to ingest the finding aid.

## Component Element and Date Handling

The handling of two specific aspects of EAD—component element conventions and date formatting—is central to ensuring successful imports into CONTENTdm. These features relate directly to the granularity of the archival description and the machine-readability of temporal information, respectively. Analysis of an institutional EAD example and examination of community-reported issues provide clear guidance on CONTENTdm's expectations. The system demonstrates flexibility in its acceptance of different component numbering schemes while simultaneously showing a sophisticated approach to date management that balances human-readable text with machine-parsable formats. This dual capability is a hallmark of a mature digital repository system designed for both usability and powerful backend searching and sorting.

Regarding component element conventions, the choice between generic `<c>` elements (e.g., `<c level="item">...</c>`) and numbered conventions like `<c01>`, `<c02>`, etc., is a long-standing debate in the archival community. The research goal was to determine if CONTENTdm had a preference or a strict requirement for one over the other. The EAD file from Columbia University Libraries, which is successfully processed by CONTENTdm, utilizes the generic `<c>` convention, employing an `@id` attribute (`id="p1"`) for identification [[14](https://findingaids.library.columbia.edu/ead/cul-10491409.xml)]. This single example is highly significant because it proves that the numbered `<c0n>` convention is not a strict requirement. If CONTENTdm's parser were rigidly tied to the numbered scheme, it would reject or fail to properly interpret the Columbia EAD, leading to widespread complaints in community forums. The absence of such complaints in the provided search results for Code4Lib and other platforms suggests that the system's parser is robust and can accommodate either convention.

This flexibility is a practical advantage for institutions. Many archival description tools and workflows default to the generic `<c>` element with a `@level` attribute, as it is simpler to generate programmatically and decouples the element's presentation from its structural role. The fact that CONTENTdm accepts this method means that developers of repository presets, like Cartulary, can confidently default to the generic `<c>` convention without fear of import failure. The `strictComponentConvention: false` setting in the proposed preset configuration is therefore strongly justified, as enforcing a numbered convention would unnecessarily restrict the usability of the preset for a large portion of the archival community. The system's ability to parse and render descriptions regardless of the component element's name indicates a focus on semantic meaning (the `@level` attribute) over syntactic form (the element's prefix).

On the topic of date handling, CONTENTdm exhibits a notable sophistication. The `<unitdate>` element in the Columbia University EAD provides a perfect case study [[14](https://findingaids.library.columbia.edu/ead/cul-10491409.xml)]. The element contains the free-text date string "1923-2010". This immediately tells us that CONTENTdm's importer does not enforce a strict, single format for dates. Many systems mandate ISO 8601 (e.g., `1923/1924` or `1923-01-01`), and failing to comply would result in an import error. The successful import of "1923-2010" shows that CONTENTdm can handle common textual representations of date ranges. However, the element in the same example also carries a `@normal` attribute with the value `1923/2010` [[14](https://findingaids.library.columbia.edu/ead/cul-10491409.xml)]. This attribute is the key to understanding CONTENTdm's hybrid approach. The `@normal` attribute is a standardized way to provide a machine-readable equivalent of the date, typically in a format suitable for sorting and querying. The forward slash in the normal value is a common convention for representing inclusive date ranges in normalized form.

This dual-format usage—free-text for display and normalized for processing—is a powerful design choice. It allows archivists to enter dates in a way that is natural and descriptive for end-users ("1923-2010") while providing the system with a precise value for computational tasks like chronological sorting, faceted browsing, and range queries. This confirms that CONTENTdm leverages the full power of the EAD standard's date encoding capabilities. The presence of the `@normal` attribute in a working example is definitive proof that the system not only accepts but expects this attribute for proper date management. While it may fall back on parsing the content of the `<unitdate>` element if the `@normal` attribute is absent, relying on it is the correct and most robust practice. This finding has direct implications for the validationStrictness setting in the preset configuration. Since free-text dates are accepted, a strict enforcement of ISO 8601 alone would be incorrect. The `isoDate` validation should be set to `'warning'`, encouraging the use of the `@normal` attribute for better functionality without blocking imports that lack it.

The handling of composite dates, such as start/end ranges, is also supported through this mechanism. While the provided EAD only shows an inclusive range (`type="inclusive"`), the use of the `@normal` attribute for bulk dates (`type="bulk"`) suggests the system can manage different date types [[14](https://findingaids.library.columbia.edu/ead/cul-10491409.xml)]. The `@normal` attribute can contain multiple dates separated by a delimiter (often a slash or semicolon) to represent a range or a list of distinct dates. Although the provided sources do not contain an example of a complex composite date with a `@standarddates` attribute, the existence of the `@normal` attribute in a fully functional `<unitdate>` element is the most important finding. It establishes that CONTENTdm's importer is designed to work with the EAD standard's advanced date encoding features. This robust handling of dates is a significant strength of the platform, enabling rich temporal discovery of collections.

## Character Encoding and Digital Object Integration

Two final, yet critically important, aspects of the EAD import profile for CONTENTdm concern the handling of text-based data and the integration of digital surrogates. These functionalities address the fundamental requirements of a modern digital repository: supporting multilingual and international content, and connecting descriptive information to the digitized items it represents. The analysis of an institutional EAD example and logical inference from XML processing fundamentals provide clear answers to these questions. CONTENTdm expects UTF-8 encoded files and robustly supports the linking of digital objects via the `<dao>` element, demonstrating its commitment to both linguistic inclusivity and the creation of interconnected, resource-rich finding aids.

The question of character encoding is paramount in an increasingly globalized digital landscape. The provided source materials do not contain any explicit statement from OCLC regarding the required character encoding for EAD imports. However, this absence of specific information is highly telling when viewed in the context of modern web and data standards. All contemporary XML parsers and systems default to and expect UTF-8 encoding. This is the universal standard for web content and is capable of representing virtually every character from every language, including those with complex diacritics and non-Latin scripts [[8](https://stackoverflow.com/questions/37716174/python-how-to-store-decoded-unicode-string-from-email-in-list-and-then-use-later)]. To mandate a different encoding, such as ISO-8859-1 or a legacy code page, would create significant barriers for institutions working with international collections and would lead to a high volume of cryptic import errors related to character corruption. While the provided materials contain numerous Stack Overflow questions about handling special characters like ampersands (`&`) and diacritics in various programming contexts [[5](https://stackoverflow.com/questions/18549726/element-link-is-missing-required-attribute-property), [8](https://stackoverflow.com/questions/37716174/python-how-to-store-decoded-unicode-string-from-email-in-list-and-then-use-later)], there are no reports of such issues being a problem during EAD imports into CONTENTdm. This lack of negative feedback strongly suggests that the system's XML parser is robust and handles standard XML entity encoding correctly (e.g., `&amp;`, `&lt;`). Furthermore, the placeholder configuration in Cartulary included a `nokogiriSanitize: false` setting, which implies that sanitization was not considered necessary for CONTENTdm, unlike for other systems where malformed XML entities could cause truncation errors [[24](https://community.adobe.com/questions-611/problem-adding-books-to-my-new-e-reader-399622)]. Based on this collective evidence, it is a safe and logical conclusion that CONTENTdm expects and correctly processes UTF-8 encoded EAD files. Any developer preparing files for CONTENTdm should ensure their output is saved in UTF-8 without a Byte Order Mark (BOM), which can sometimes interfere with XML processing.

The second critical function is the ability to link a finding aid to its corresponding digital objects. This is achieved in EAD through the `<dao>` (Digital Access Object) element. The presence and usage of this element in an institutional EAD file is perhaps the most direct evidence of its support. The EAD from Columbia University Libraries contains a `<dao>` element within a `<c>` component, linking to a digital object representation of a specific item in the collection [[14](https://findingaids.library.columbia.edu/ead/cul-10491409.xml)]. This example is invaluable because it not only confirms that `<dao>` is supported but also provides insight into the specific attributes that are likely required for the link to be successfully created and rendered. The `<dao>` element in the Columbia EAD includes three attributes: `@href`, `@title`, and `@role`.

The `@href` attribute is self-explanatory; it contains the URL pointing to the digital object, which in a CONTENTdm context would be the URL of the item's viewer page. The `@title` attribute provides a label for the link, such as "View digital object," which is crucial for accessibility and user comprehension. The `@role` attribute is particularly important as it semantically describes the nature of the link, for example, "use original" or "read online." This attribute helps CONTENTdm and end-users understand the relationship between the described physical item and its digital surrogate. The fact that all three attributes are present in a working example leads to a strong conclusion: for a `<dao>` to be processed correctly, `@href`, `@title`, and `@role` are likely the minimum required attributes. This finding directly informs the preset configuration, justifying the setting `requiresTitleOnDao: true`. While the provided materials do not contain examples of `<daogrp>` (to group multiple `<dao>`s) or `<daodesc>` (to provide a description of a group of links), the lack of mention of these elements suggests they may not be supported or are rarely used in the community's workflows for CONTENTdm. The primary and proven method for linking digital objects is the individual `<dao>` element with the specified attributes.

In synthesis, CONTENTdm's profile in these two areas is modern and robust. It correctly handles the complexities of international character sets by expecting UTF-8 encoding, removing a common source of frustration for global institutions. Simultaneously, it fully embraces the principle of linked data by providing a clear and well-documented pathway for embedding hyperlinks to digital objects directly within the archival description. This integration transforms the finding aid from a static document into a dynamic portal, enhancing the user experience and maximizing the utility of the digitized collection. The evidence strongly supports a configuration that generates UTF-8 files and creates `<dao>` elements with `href`, `title`, and `role` attributes.

However, a careful analysis of the few related error messages and general XML processing issues mentioned in the sources allows for a reasoned assessment of potential quirks and the appropriate level of validation strictness for a repository preset.

A comprehensive search of the provided materials for terms like "CONTENTdm EAD import," "contentdm finding aid," "import failed," and similar phrases did not uncover dedicated threads or articles focused on EAD import errors. This is a strong positive signal. Given this lack of specific community-reported EAD issues, we must turn to the evidence from the one available institutional EAD example from Columbia University [[14](https://findingaids.library.columbia.edu/ead/cul-10491409.xml)]. This file is a model of good EAD practice. It uses the correct EAD 2002 namespace, includes a `@normal` attribute on its `<unitdate>` elements, and contains all the required core elements. The fact that this example works demonstrates that meeting these standard expectations is sufficient for a successful import. This reinforces the conclusion that CONTENTdm's importer is designed to be compliant with the EAD 2002 standard rather than imposing a unique, non-standard dialect. The lack of community complaints about quirky behaviors, such as requiring non-standard attributes or misinterpreting element names, further supports this conclusion.

This leads to a logical approach for configuring the validation strictness in a repository preset. The goal is to balance encouraging best practices with avoiding unnecessary import failures. Based on the evidence:

*   **ISO 8601 Dates (`isoDate`):** As established previously, the importer accepts free-text dates and relies on the `@normal` attribute for machine processing. Enforcing strict ISO 8601 validation would break compatibility with the common practice of using free-text ranges like "1923-2010." Therefore, the validation should be set to `'warning'`, alerting the user that a `@normal` value would improve functionality without causing an import to fail.
*   **Duplicate Unit IDs (`duplicateUnitid`):** This validation should be set to `'error'`. While not explicitly stated, the purpose of a `<unitid>` is to provide a unique identifier for the collection. Having duplicates would create ambiguity and likely lead to broken links or incorrect associations within the CONTENTdm system. This is a strong candidate for a hard validation rule.
*   **Extent Format, Chronological Order, and Internal Unit ID Audience:** For these validation rules, the provided sources contain no information. They were present in the original placeholder configuration but their applicability to CONTENTdm is unknown. In the absence of evidence, the safest approach is to set them to `'warning'`, assuming they are desirable best practices but not likely to cause a catastrophic import failure. This prevents the preset from being overly restrictive without a basis in documented system behavior.

Regarding version-specific differences, the provided context does not contain release notes or community discussions comparing the EAD import capabilities of CONTENTdm versions 6, 7, and 8. The analysis is therefore based on the structure of an EAD file from a current, live repository, which is most likely running a recent version. The profile should be targeted at the latest stable version, as this is the system most institutions are using today. Significant changes in EAD support between minor versions are unlikely, but without specific documentation, this remains an assumption. The recommendation to test on a trial instance is particularly relevant here to confirm that the profile holds up across different environments.

Finally, one significant open question that arises from this analysis is how CONTENTdm handles controlled vocabulary fields, such as `<subject>`, `<genreform>`, and `<name>` with `@role`. The institutional EAD example does not contain extensive examples of these elements, and the community sources are silent on the matter [[14](https://findingaids.library.columbia.edu/ead/cul-10491409.xml)]. It is unclear whether these fields are imported as plain text for display only, or if CONTENTdm indexes them to create searchable facets. This is a critical piece of information for maximizing the discoverability of collections. A future investigation should prioritize resolving this question, as it has a major impact on the utility of the imported metadata.

## Synthesis and Preset Configuration Recommendations

This deep research report sought to establish an evidence-based EAD import profile for OCLC's CONTENTdm, moving from uninformed placeholders to a configuration grounded in analysis of institutional examples and community knowledge. The investigation systematically addressed each key aspect of the import process, from the overarching EAD standard to the specifics of character encoding and digital object linking. The findings, derived primarily from a single but highly informative institutional EAD example and the conspicuous absence of negative community reports, paint a clear picture of CONTENTdm as a system that prioritizes interoperability through adherence to the EAD 2002 standard. It is a mature and stable platform that expects well-formed, UTF-8 encoded XML files with a predictable structural hierarchy. This section synthesizes the key findings and translates them into a definitive, actionable preset configuration for Cartulary, along with a list of remaining open questions that require empirical validation.

The core determination of this research is that CONTENTdm's EAD importer is designed for **EAD 2002**. This conclusion is firmly based on the observation of the EAD 2002 namespace (`http://www.loc.gov/ead/`) in a functional finding aid hosted within the system [[14](https://findingaids.library.columbia.edu/ead/cul-10491409.xml)]. The system demonstrates flexibility in its parsing, accepting both generic `<c>` elements and numbered `<c0n>` conventions, making the generic form a perfectly valid choice for a repository preset [[14](https://findingaids.library.columbia.edu/ead/cul-10491409.xml)]. The importer's sophistication is particularly evident in its date handling; it gracefully manages both free-text date expressions (like "1923-2010") and machine-readable equivalents provided via the `@normal` attribute, indicating a hybrid approach that serves both user display and backend computational needs [[14](https://findingaids.library.columbia.edu/ead/cul-10491409.xml)]. Furthermore, the system robustly supports the linking of digital objects through the `<dao>` element, requiring at a minimum the `@href`, `@title`, and `@role` attributes for the links to be successfully processed [[14](https://findingaids.library.columbia.edu/ead/cul-10491409.xml)]. The overall evidence suggests a system that is largely free of the quirky, non-standard behaviors seen in some other platforms, relying instead on solid XML and EAD 2002 compliance.

Based on this comprehensive analysis, the following `PresetConfig` object is recommended for CONTENTdm. Each property is justified by the evidence presented throughout this report.

```typescript
contentdm: {
  name: 'contentdm',
  label: 'CONTENTdm',
  targetFormat: 'ead2002',
  componentConvention: 'generic-c',
  strictComponentConvention: false,
  validationStrictness: {
    isoDate: 'warning',
    extentFormat: 'warning',
    chronologicalOrder: 'warning',
    unitidAudienceInternal: 'warning',
    duplicateUnitid: 'error',
  },
  nokogiriSanitize: false,
  requiresTitleOnDao: true,
},
```

*   **`targetFormat: 'ead2002'`**: Justified by the EAD 2002 namespace found in a live institutional example [[14](https://findingaids.library.columbia.edu/ead/cul-10491409.xml)]. Support for EAD3 is not confirmed.
*   **`componentConvention: 'generic-c'`**: Justified by the use of generic `<c>` elements in a working EAD, proving it is a sufficient and accepted convention [[14](https://findingaids.library.columbia.edu/ead/cul-10491409.xml)].
*   **`strictComponentConvention: false`**: Justified by the evidence that numbered conventions are also accepted, making a strict enforcement of generic `<c>` unnecessary and potentially burdensome.
*   **`validationStrictness.isoDate: 'warning'`**: Justified by the successful import of a free-text date range, indicating that while the `@normal` attribute is beneficial, strict ISO 8601 is not enforced [[14](https://findingaids.library.columbia.edu/ead/cul-10491409.xml)].
*   **`validationStrictness.duplicateUnitid: 'error'`**: Recommended based on the inferential logic that a unique `<unitid>` is a core requirement for data integrity, even though this is not explicitly documented.
*   **`nokogiriSanitize: false`**: Justified by the lack of community reports about XML parsing issues and the robust nature of the system, suggesting that standard UTF-8 encoded XML is sufficient [[24](https://community.adobe.com/questions-611/problem-adding-books-to-my-new-e-reader-399622)].
*   **`requiresTitleOnDao: true`**: Justified by the presence of the `@title` attribute on the `<dao>` element in a working institutional example, implying it is a required attribute for the link to be processed [[14](https://findingaids.library.columbia.edu/ead/cul-10491409.xml)].

Despite the clarity of these recommendations, several open questions remain, stemming from the inherent limitations of secondary-source research. Resolving these will require direct, hands-on testing.

| Open Question | Recommendation for Resolution |
| :--- | :--- |
| **Absolute Minimum Field Set:** What is the precise minimum requirement? Does the import fail if `<unitdate>` is omitted? | Create a series of test EAD files, progressively removing `<unitdate>`, then `<did>`, and observing the import result. |
| **Namespace Requirement:** Is the `xmlns` declaration on the `<ead>` root element strictly required, or is schema validation enough? | Generate an EAD file that is well-formed and validates against the EAD 2002 DTD/XSD but omits the `xmlns` declaration. Attempt to import it. |
| **Schema Location (`xsi:schemaLocation`):** Is a `xsi:schemaLocation` attribute required? If so, what URL should be used? | Test importing an EAD with no `xsi:schemaLocation`, then one with a standard EAD 2002 URL, to determine if it is mandatory and which one the system expects. |
| **`@audience` Attribute:** Is the `@audience` attribute on the `<ead>` element required? | Create two identical EAD files, one with a `@audience="external"` attribute and one without, and test their import behavior. |
| **Controlled Vocabulary Fields:** How are `<subject>`, `<genreform>`, and `<name>` elements handled? Are they indexed for searching or just displayed as text? | Populate these fields in a test EAD and then check within the CONTENTdm interface to see if they create searchable facets or browse options. |
| **Maximum Nesting Depth:** Is there a limit to the nesting depth of `<c>` elements? | Create an EAD with a deeply nested collection structure (e.g., 20 levels deep) to test if the system imposes a recursion limit. |
| **Composite Date Support:** Is there explicit support for composite date ranges using the `@standarddates` attribute on `<unitdate>`? | Use the `@standarddates` attribute in a test `<unitdate>` and verify if the system correctly interprets and displays the multiple dates. |

One uses the generic `<c>` convention, and the other uses the numbered `<c0n>` convention, both modeled after the structure of the institutional example analyzed in this report [[14](https://findingaids.library.columbia.edu/ead/cul-10491409.xml)].
