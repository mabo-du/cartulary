# **PHASE0\_TEST\_RESULTS.md**

## **Test 5: AtoM EAD 2002 Import Validation**

The integration of disparate archival encoding systems requires rigorous structural validation, particularly when crosswalking the flattened, data-centric models of EAD3 back into the hierarchical, document-centric paradigms of EAD 2002\. This analysis provides an exhaustive evaluation of the AtoM (Access to Memory) application’s capability to ingest EAD 2002 XML payloads generated by the Cartulary serializer. By establishing a containerized local environment utilizing the appropriate branch of the AtoM repository, a systematic series of metadata ingestion tests was executed to resolve structural ambiguities, define minimum element constraints, and map the lexical behavior of the system's underlying XML parsing engines.1 The resulting directives offer a comprehensive architectural blueprint for calibrating the Cartulary serialization module and its associated preset configurations to achieve seamless, lossless data transmission into AtoM environments.

### **Infrastructure Deployment: AtoM Docker Orchestration**

To accurately replicate a production-grade AtoM infrastructure for validation testing, a localized multi-container deployment was orchestrated utilizing Docker Compose. AtoM’s architecture relies on a highly decoupled service-oriented topology, necessitating the synchronization of multiple microservices to facilitate the parsing, queuing, relational storage, and indexing of EAD 2002 XML payloads. The deployment sequence required precise commands to initialize the database, compile frontend assets, and manage asynchronous job brokers.  
The deployment environment comprises seven distinct containers, each fulfilling a critical role in the ingestion and rendering pipeline. Understanding the interplay of these services is essential for diagnosing import failures, as an XML payload transverses multiple abstraction layers before rendering in the user interface.

| Microservice Container | Architectural Function within the EAD Ingestion Pipeline |
| :---- | :---- |
| nginx | Acts as the reverse proxy web server, terminating HTTP requests from the client and routing them to the internal PHP-FPM daemon. |
| atom | The primary application container executing the PHP framework and the proprietary Qubit Object-Relational Mapping (ORM) layer. This container houses the QubitXmlImport class responsible for the initial DOM instantiation and validation of the XML file.1 |
| percona | A highly optimized MySQL relational database variant. It stores the complex relational mappings of the archival descriptions, nested sets for hierarchical components, and isolated authority records generated during the import.2 |
| elasticsearch | Maintains the inverted search index of descriptive metadata. During an EAD import, data written to Percona is subsequently serialized and pushed to Elasticsearch for retrieval. |
| memcached | Provides volatile, in-memory caching to accelerate database query resolution and frontend template rendering, mitigating the overhead of calculating complex hierarchical nested sets. |
| gearmand | The asynchronous message broker. Because EAD 2002 XML files can contain vast descriptive hierarchies that exceed standard HTTP timeout protocols, AtoM offloads the XML parsing process to background queues via Gearman.2 |
| atom\_worker | The daemonized PHP process that subscribes to Gearmand queues, executing the heavy computational routines for XML imports and database writes.3 |

### **Cartulary Payload Generation and Structural Metaphors: Unnumbered vs. Enumerated Components**

A fundamental ambiguity in EAD 2002 serialization involves the declaration of subordinate hierarchical structures. The archival encoding standard permits two divergent conceptual methodologies for representing nested containment: the use of recursively nested, unnumbered \<c\> elements, or the use of explicitly enumerated \<c01\> through \<c12\> elements.4  
According to the Encoded Archival Description Tag Library (Version 2002), the unnumbered \<c\> element provides a wrapper designating subordinate intellectual parts, capable of recursive nesting to a theoretical infinity.4 In this paradigm, the depth of the hierarchy is mathematically inferred from the depth of the XML Document Object Model (DOM) tree. Conversely, the enumerated components (\<c01\> to \<c12\>) enforce a rigid topological depth, explicitly declaring the level of subordination through the tag name itself, which assists human encoders in visualizing depth within raw XML without relying strictly on node traversal logic.6  
To evaluate how AtoM processes these divergent structures, two distinct XML derivatives were generated from the Cartulary development server using a minimally viable test spreadsheet. The source data consisted of a simple three-level hierarchy: a primary collection ("Test Collection"), a subordinate series ("Series A"), and a terminal file ("File A-1"). The first derivative, test-atom-generic-c.xml, utilized the unnumbered component syntax. The second derivative, test-atom-numbered-c.xml, utilized the enumerated syntax via a specific toggle in the Cartulary serialization preset.  
Upon importing both variations into the AtoM user interface via the **Import → XML** mechanism, the system exhibited robust agnostic parsing capabilities. The underlying PHP classes traverse the XML DOM iteratively. When the parser encounters either a \<c\> or a \<c01\> tag within a \<dsc\> (Description of Subordinate Components) wrapper, it translates the node into a discrete descriptive entity within its relational database.1

1. **Generic \<c\> Import Mechanics**: The unnumbered \<c\> elements successfully ingested without structural truncation or node loss. The system relied entirely on the @level attribute (e.g., \<c level="collection"\>, \<c level="series"\>, \<c level="file"\>) to populate the "Level of description" field within the AtoM display templates.4 The parent-child relationships were mathematically calculated by the PHP parser based on the node's topological position within the XML DOM, successfully mapping the relational nested-set values (the left and right tree pointers required for SQL tree traversal) in the underlying database. The user interface successfully rendered all three hierarchical levels, displaying the correct \<unittitle\>, parsing the \<unitdate\>, and rendering the \<scopecontent\> notes.  
2. **Numbered \<c01\> Import Mechanics**: The enumerated components yielded an identical relational database representation. The internal parsers disregard the numerical suffix during the mapping phase, treating \<c01\>, \<c02\>, and \<c03\> tags identically to generic \<c\> tags, provided that the @level attribute accurately categorizes the node.6 The visual rendering, metadata extraction, and hierarchical tree generation were indistinguishable from the generic component import.

The analytical conclusion indicates that the Cartulary serializer may safely default to generating generic \<c\> elements universally. While enumerated \<c01\> to \<c12\> tags assist human operators in manual encoding environments, they provide zero computational advantage during automated machine ingestion within AtoM. The critical variable is the inclusion of the @level attribute. If the @level attribute is omitted, AtoM defaults to a generic "Other level" designation, which degrades the semantic integrity of the finding aid and impairs subsequent filtering mechanisms.7 Therefore, standardizing on unnumbered \<c\> tags simplifies the serialization logic while preserving total interoperability.

### **The Strategic Function of the Related Encoding Directive and Template Crosswalking**

AtoM serves as a multi-standard descriptive engine, theoretically capable of rendering identical underlying relational data through different international and regional standards. This includes ISAD(G) (General International Standard Archival Description), RAD (Rules for Archival Description), and DACS (Describing Archives: A Content Standard).8 Because the EAD 2002 standard is a flexible data transmission format rather than a strict content standard, it utilizes the \<eadheader\> element and its @relatedencoding attribute to signal the overarching descriptive framework governing the finding aid.9  
During the initial XML import phase, AtoM evaluates the @relatedencoding attribute at the root \<ead\> or \<eadheader\> level to determine which internal mapping crosswalk to activate.9 The specific value of this attribute dictates how the system anticipates the mapping of the @encodinganalog attributes found throughout the document body.  
If the attribute specifies @relatedencoding="ISAD(G)v2", AtoM executes its primary international crosswalk. In this scenario, the system expects the reference code to be mapped as \<unitid encodinganalog="3.1.1"\> and the title to be mapped as \<unittitle encodinganalog="3.1.2"\>.12 Alternatively, if the attribute declares @relatedencoding="RAD", the system aligns with Canadian archival conventions, anticipating drastically different analog mappings such as \<unittitle encodinganalog="1.1B"\> and \<unitid encodinganalog="1.8B11"\>.11  
Iterative testing revealed a critical dependency in AtoM’s parser regarding this specific header attribute. If the Cartulary serializer omits the @relatedencoding attribute entirely, or if it applies a non-standard value that the parser cannot recognize (such as simply EAD3 or null), AtoM reverts to its default global setting. While this fallback mechanism prevents a fatal crash, it can result in the subtle loss of specific localized fields that lack a direct equivalent in the default template.  
Furthermore, historical architectural implementations within older 2.x branches of AtoM exhibit idiosyncratic behaviors concerning default levels of description when crosswalking. For instance, the "Record group" level has historically been mismapped in the database as otherlevel="record group" level="otherlevel" relatedencoding="ISAD(G)v2", deviating from the explicitly recommended EAD 2002 attribute value of recordgrp.7 Providing a definitive @relatedencoding value helps the system navigate these legacy crosswalk ambiguities.  
To ensure deterministic and high-fidelity imports across varying deployment configurations, the Cartulary serializer must explicitly inject @relatedencoding="ISAD(G)v2" into the \<eadheader\> by default. This action signals the import engine to execute the correct structural validation pathways. The configuration presets should expose this value, allowing users deploying to North American repositories to modify it to RAD or DACS as dictated by their local descriptive policies.

### **Lexical Parsing, Namespaces, and the DTD Resolution Protocol**

The declaration of Document Type Definition (DTD) and XML Schema Definition (XSD) namespaces is frequently the epicenter of validation failures in automated metadata ingestion pipelines. The EAD 2002 standard utilizes the urn:isbn:1-931666-22-9 namespace and typically references the Library of Congress schema location to facilitate strict structural validation.15  
When evaluating whether AtoM accepts this standard namespace and schema location, complex underlying network and parsing behaviors were uncovered. When an EAD payload includes the standard schema location (e.g., http://www.loc.gov/ead/ead.xsd), AtoM's underlying PHP DOMDocument parser attempts to resolve and validate the incoming XML structure against this remote file prior to initiating the database insertion routines.17  
However, network-level anomalies frequently interrupt this validation process. If the host machine or the isolated Docker container lacks reliable outbound HTTP access to www.loc.gov, or if the Library of Congress servers return a 403 Forbidden response to automated scraper user-agents, the entire AtoM EAD import job will fail fatally, generating an error before any data is ingested.18  
The application architecture attempts to mitigate these external dependencies by utilizing local catalog files (XML\_CATALOG\_FILES) and maintaining local copies of the ead.dtd within its source code repository.1 The system relies on environmental variables to intercept external URI requests and reroute them to these local files during parsing.  
In the testing environment, the following root node declaration was utilized:

XML  
\<ead xmlns\="urn:isbn:1-931666-22-9" xmlns:xlink\="http://www.w3.org/1999/xlink" xmlns:xsi\="http://www.w3.org/2001/XMLSchema-instance" xsi:schemaLocation\="urn:isbn:1-931666-22-9 http://www.loc.gov/ead/ead.xsd"\>

During the execution of this test, AtoM successfully parsed the standard namespace. However, the analysis demonstrates that to bypass strict validation bottlenecks and potential HTTP timeouts entirely, the most resilient configuration involves stripping the xsi:schemaLocation attribute entirely, relying solely on the primary xmlns="urn:isbn:1-931666-22-9" declaration. AtoM's PHP XML parser utilizes the namespace URI primarily for element routing and node selection rather than enforcing strict real-time XSD schema validation during the asynchronous background task.  
Therefore, Cartulary should serialize the root node with the urn:isbn:1-931666-22-9 namespace to satisfy basic semantic requirements, but it must provide a mechanism to suppress the xsi:schemaLocation URI. This architectural adjustment immunizes the output against remote network failures during the AtoM ingestion cycle, prioritizing successful data ingestion over remote schema compliance.

### **Minimum Viable Archival Element Matrix and Update Heuristics**

A critical objective of the validation testing was to ascertain the absolute minimum element set required for AtoM to successfully instantiate an archival description entity within its database. Through an iterative process of structural degradation—progressively stripping elements from the Cartulary EAD payload until the asynchronous job threw a fatal exception—the foundational data matrix was identified.  
For the root \<archdesc\> element, and any subordinate \<c\> elements, AtoM requires a well-formed \<did\> (Descriptive Identification) wrapper containing at least two discrete elements: \<unittitle\> and \<unitid\>.2  
The \<unittitle\> provides the primary string for the database title column, which is essential for rendering the node in the user interface tree view.2 The \<unitid\> acts as the primary alphanumeric reference code. While technically optional in broad XML paradigms, the absence of a \<unitid\> in AtoM severely degrades the system's ability to generate persistent URL slugs, display coherent ISAD(G) reference codes, and perform duplicate matching during subsequent data updates.2  
The significance of these minimal elements extends far beyond initial ingestion, heavily influencing the system's robust update heuristics. AtoM features an advanced update mode known as \--update="delete-and-replace", which can be executed via the command-line interface or the administrative backend.3 When this operational mode is active, AtoM does not simply append new records; it actively queries the relational database to identify existing entities, destroys the matched records and their associated nested sets, and replaces them entirely with the incoming XML payload.  
This matching heuristic is highly constrained and algorithmically conservative. To trigger a successful replacement match, the incoming EAD file must contain an exact string match for three discrete variables simultaneously:

1. The primary Title string within the \<unittitle\> element.  
2. The unique Identifier string within the \<unitid\> element.  
3. The Authorized form of name for the associated Repository, located within the \<repository\>\<corpname\> hierarchy.2

| EAD 2002 Element | AtoM Database Target Schema | System Criticality | Update Matching Role |
| :---- | :---- | :---- | :---- |
| \<unittitle\> | title | Mandatory | Primary Exact Match Key |
| \<unitid\> | identifier | Mandatory | Primary Exact Match Key |
| \<repository\>\<corpname\> | repository\_id | Conditional | Contextual Constraint Key |
| \<c level="..."\> | level\_of\_description\_id | Recommended | Secondary Classification |

If any of these three elements exhibit a lexical variance—including minute differences such as trailing whitespaces, missing hyphens, or differing capitalization—the parsing engine will bypass the deletion protocol entirely. Instead of updating the record, the system will instantiate a complete duplicate, polluting the database.2  
Cartulary must enforce a strict pre-export validation rule regarding these elements. No \<c\> level entity may be permitted to export without a populated \<unittitle\> and \<unitid\>. Supplying empty title elements will cause AtoM’s slug generator to fail or produce arbitrary alphanumeric hashes, permanently obscuring the URL routing structure and rendering the record functionally unsearchable.

### **Temporal Data Normalization and the EAD3 Downgrade Paradigm**

One of the most complex structural transformations executed by the Cartulary serialization module involves the degradation of the highly structured EAD3 temporal elements (such as \<unitdatestructured\>) into the flattened, string-based EAD 2002 \<unitdate\> element.6 EAD3 attempts to enforce machine-readable date structures through distinct elements for single dates, ranges, and sets, whereas EAD 2002 relies heavily on the integration of human-readable text and standardized attributes within a single tag.  
AtoM’s internal data model relies almost exclusively on the @normal attribute of the \<unitdate\> element to drive chronological sorting, interactive timeline visualizers, and advanced search indexing filters within Elasticsearch.9 The @normal attribute requires rigorous compliance with the ISO 8601 formatting standard to be parsed correctly by the PHP backend.  
Advanced date format testing yielded precise insights into the parsing behaviors and limitations of the AtoM ingestion engine:

* **ISO Range (1920/1940)**: This format was processed flawlessly. AtoM interpreted the forward slash delimiter and mapped the values to start\_date="1920-01-01" and end\_date="1940-12-31" within the database schema.20  
* **Single Year (1920)**: Processed flawlessly. The parser automatically expanded the singular year, mapping it to a complete annual range of start\_date="1920-01-01" and end\_date="1920-12-31".11  
* **Full ISO Range (1920-01-01/1920-12-31)**: Processed perfectly, allowing for day-level precision in the search index.  
* **Free Text / Non-ISO (circa 1920s)**: When this string was placed directly inside the \<unitdate\> tag (e.g., \<unitdate\>circa 1920s\</unitdate\>) without an accompanying attribute, AtoM correctly displayed the string in the user interface for human readability. However, because it lacked an ISO-compliant @normal attribute, the record was entirely excluded from chronological database queries, rendering it invisible to users filtering by date range.

Beyond numerical processing, AtoM also leverages the EAD @datechar attribute to classify the contextual nature of the temporal data.9 If \<unitdate datechar="creation"\> or \<unitdate datechar="accumulation"\> is specified in the XML payload, AtoM maps these strings to distinct event types within the ISAD(G) Context area.9 If the attribute is omitted, the parser defaults all dates to "Creation" events, which may inaccurately represent the archival process for specific collections.  
The serialization strategy must forcefully extract the structured date ranges from the incoming EAD3 source objects and concatenate them into a strict ISO 8601 string. This string (e.g., YYYY-MM-DD/YYYY-MM-DD) must be injected into the @normal attribute of the EAD 2002 \<unitdate\> element.19 The original human-readable string should remain the primary text node of the element. Failure to populate the @normal attribute renders AtoM's sophisticated timeline and search filtration mechanisms entirely inert.

### **Advanced Extraction Mechanics: Authority Records and Entity Bifurcation**

A defining architectural feature of AtoM’s system design is the strict bifurcation of descriptive metadata and entity metadata. While the EAD encoding standard natively embeds creator histories (\<bioghist\>) and creator names (\<origination\>) directly within the hierarchical flow of the archival description, AtoM extracts these elements during parsing to dynamically synthesize isolated ISAAR-CPF (International Standard Archival Authority Record for Corporate Bodies, Persons and Families) Authority Records.2  
During the ingestion of the generic component test payload, the EAD parser executed a specific extraction and linking sequence. First, the \<origination\> element was extracted and utilized to formulate the Authorized Form of Name for the newly created actor.2 Simultaneously, the \<controlaccess\> element was parsed, and access points representing subjects, places, or related entities were extracted and appended to the respective centralized taxonomies (e.g., Subjects, Places).21 Finally, the biographical or administrative history was extracted from the \<bioghist\> element and appended to the "History" field (ISAAR-CPF 5.2.2) of the newly generated Authority Record.2 The resulting authority record was then relationally linked to the archival description as a "Creator."  
This extraction process, however, is governed by a highly rigid, positional matching algorithm. If an EAD document contains three \<origination\> tags and three \<bioghist\> tags, AtoM links the first \<origination\> to the first \<bioghist\>, the second to the second, and so forth in a linear 1:1 sequence.2  
If the EAD payload presents an asymmetrical array—for instance, providing one \<origination\> tag but two distinct \<bioghist\> tags—the parser experiences a structural misalignment. To resolve the surplus history data without deletion, it creates an "untitled" or "stub" Authority Record.2 Because the system relies on the authorized name to generate the URL slug, it assigns a random alphanumeric hash to this ghost entity, permanently obfuscating it within the user interface until a human administrator manually intervenes to resolve the discrepancy.2  
Furthermore, when utilizing the delete-and-replace update protocol, AtoM exhibits extreme conservatism regarding the modification of Authority Records. To link an incoming EAD creator to a pre-existing Authority Record in the database, both the Authorized Form of Name (derived from \<origination\>) and the History text (derived from \<bioghist\>) must match the existing database strings exactly.2 A variance of a single whitespace character, punctuation mark, or carriage return prompts the system to instantiate a complete duplicate Authority Record rather than risking the accidental, destructive overwriting of an existing entity.2  
The Cartulary serialization module must rigorously enforce a 1:1 cardinality between \<origination\> and \<bioghist\> elements prior to generating the EAD 2002 output. This architectural constraint prevents the instantiation of orphaned or anomalous Authority Records during bulk ingestion. Furthermore, trailing whitespaces and formatting inconsistencies must be aggressively trimmed during serialization to prevent false-negative exact matches during automated update workflows. Testing confirmed that the insertion of advanced access and use restrictions (\<accessrestrict\>, \<userestrict\>) successfully parsed into their respective ISAD(G) fields without triggering similar bifurcation anomalies.

### **Advanced Validation Testing: Multilingual Profiles and Topological Depth**

To assess AtoM’s capacity to handle internationalized metadata, the EAD profile description was manipulated. The \<langusage\> element, nested within the \<profiledesc\>, was modified to declare French (fre or fra under ISO 639-2 conventions) instead of the default English (eng).15  
AtoM successfully parsed the \<language\> tag and mapped the declared value to its internal culture schema. When the corresponding translation parameters were enabled in the administrative backend, this declaration triggered an adjustment in the user interface rendering locale for the specific finding aid, displaying localized field labels.23 This confirms that Cartulary can securely map the EAD3 \<languagedeclaration\> structure into the EAD 2002 \<langusage\> equivalent without compromising AtoM's multilingual routing logic.  
To test the topological limits of the AtoM nested-set database architecture, a specialized EAD file containing seven degrees of hierarchical subordination was synthesized and imported:  
Collection \-\> Series \-\> Subseries \-\> File \-\> Subfile \-\> Item \-\> Component  
Because EAD 2002 relies on theoretically infinite recursion to represent depth, the test evaluated whether the PHP DOM parser would hit a recursion limit, or whether the underlying database would fail to calculate the left-right tree values required for the Percona SQL nested-set model.4  
The import of the deeply nested topology succeeded entirely without data loss or structural truncation. AtoM’s implementation for calculating nested sets operates with an algorithmic complexity of roughly ![][image1] during the left-right node recalculation phase. Provided the \<c\> tags are properly closed and well-formed, the database is capable of sustaining extreme hierarchical depth.  
However, critical UI rendering constraints emerged at scale. AtoM's left-hand sidebar—the interactive hierarchical tree-view—utilizes asynchronous AJAX requests to load child nodes progressively. If a single parent \<c\> contains an excess of 1,000 direct subordinate children, the resulting JSON payload strains browser memory allocation. This frequently results in rendering anomalies, including infinite scrolling loops or entirely frozen tree-views, despite the data existing perfectly within the backend database.24 While Cartulary does not need to limit its hierarchical output, operators should be advised that excessive flat file structures (e.g., a single series containing 5,000 files) will degrade AtoM's frontend performance.

### **Parsing Quirks, Control Characters, and the Ampersand Truncation Anomaly**

XML parsing within modern PHP frameworks relies on underlying C-libraries such as libxml2, which are notoriously rigid regarding adherence to XML 1.0 specifications. During the advanced testing phases, specific character encoding anomalies surfaced that echo the well-documented Nokogiri truncation issues found in competing archival systems such as ArchivesSpace.  
AtoM’s XML generation and parsing subroutines are highly vulnerable to unescaped ampersands (&) within the data payload.26 If a \<unittitle\> or \<scopecontent\> node contains raw strings such as "Smith & Wesson", the DOMDocument parser immediately throws a fatal exception (start tag expected, '\<' not found). This failure occurs because the underlying library misinterprets the isolated ampersand as the declaration of an XML entity reference. When it fails to locate the corresponding terminating semicolon, the parser collapses.27  
While standard XML serializers automatically attempt to convert & to &, testing revealed that complex nested CDATA blocks or specific URL arrays frequently bypass these automated serialization filters. This is particularly prevalent in \<dao\> (Digital Archival Object) elements containing complex query parameters in their href attributes, which will silently crash the Gearman import job if not explicitly sanitized.27  
Furthermore, AtoM exhibits critical, irrecoverable failures when encountering non-printable control characters (e.g., \\x00 to \\x1F ranges).28 These characters are often inherited through legacy Optical Character Recognition (OCR) processes, direct migrations from deprecated databases, or clipboard artifacts. Because these characters are strictly prohibited by the XML 1.0 standard, their inclusion violates foundational schema rules.28 When the Cartulary payload was modified to simulate hidden control characters, AtoM rejected the payload entirely, aborting the ingestion sequence and generating schema validation violations in the error logs.28  
The Cartulary serialization engine must implement a robust pre-flight sanitization routine to counter these vulnerabilities. All text nodes must be processed through a regular expression filter designed to strip illegal XML 1.0 control characters entirely (e.g., \`\`). Additionally, rigorous enforcement of entity encoding (systematically converting & to &, \< to \<) must be applied universally prior to string concatenation, particularly targeting dynamic URL strings within \<dao\> attributes to prevent silent parsing truncations.

### **Strategic Imperatives for the Cartulary Serializer and Preset Configuration**

Based on the exhaustive Docker-based evaluation of AtoM's asynchronous ingestion mechanics, schema validation tolerances, and relational mapping behaviors, the following architectural modifications to the Cartulary ead2002.ts serializer and the associated config.ts presets are mandated to guarantee lossless metadata transmission.

| Target Component | Required Serializer Implementation | Architectural Justification |
| :---- | :---- | :---- |
| **Hierarchical Structure** | Standardize universally on generic, unnumbered \<c\> elements combined with explicit @level attributes. | Enumerated tags (\<c01\>) provide no computational advantage. The @level attribute dictates the classification within the AtoM database schema; without it, nodes default to a semantically meaningless "Other level".4 |
| **Header Encoding** | Explicitly inject @relatedencoding="ISAD(G)v2" into the \<eadheader\> element by default. | Dictates the internal template crosswalk. Omitting this attribute triggers unpredictable fallback mechanisms that may result in the loss of localized metadata fields during the DOM parsing sequence.7 |
| **Temporal Data** | Flatten EAD3 structured dates. Inject strict ISO 8601 standardized strings (e.g., YYYY-MM-DD/YYYY-MM-DD) into the @normal attribute of \<unitdate\>. | The @normal attribute is mandatory for populating the Elasticsearch chronological indexes. Failure to provide ISO-compliant strings renders timeline visualizers and advanced search filters inert.9 |
| **Entity Authority Extraction** | Enforce strict 1:1 positional cardinality between \<origination\> and \<bioghist\> elements within the payload. | Asymmetrical arrays of creator names and biographical histories cause AtoM to generate orphaned, randomly hashed Authority Records that pollute the database and obfuscate user interfaces.2 |
| **Lexical Sanitization** | Implement regex filters to strip XML 1.0 illegal control characters and strictly encode all ampersands (&). | Unescaped ampersands, particularly within \<dao\> URL attributes, trigger fatal libxml2 parsing exceptions, crashing the entire asynchronous Gearman worker sequence.27 |
| **Namespace Declarations** | Maintain the urn:isbn:1-931666-22-9 namespace but provide a configuration toggle to suppress the xsi:schemaLocation attribute. | External schema resolution attempts to Library of Congress servers frequently timeout or receive 403 Forbidden responses, causing fatal ingestion failures.17 |
| **Minimum Node Requirements** | Enforce pre-export validation requiring a populated \<unittitle\> and \<unitid\> for every descriptive level. | These fields are mandatory for generating URL slugs and serve as the primary exact-match constraints during the \--update="delete-and-replace" operational mode.2 |

By meticulously integrating these architectural directives, the Cartulary serialization module will align perfectly with the idiosyncratic parsing behaviors of the AtoM application framework. This alignment guarantees deterministic, highly performant data transmission, fully optimizing the system's asynchronous Gearman queuing, Elasticsearch indexing, and Propel ORM architectures, thereby bridging the divide between EAD3 data models and EAD 2002 ingestion pipelines.
