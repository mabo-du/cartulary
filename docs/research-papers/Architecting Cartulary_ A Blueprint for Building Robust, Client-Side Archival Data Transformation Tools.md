# Architecting Cartulary: A Blueprint for Building Robust, Client-Side Archival Data Transformation Tools

## Spreadsheet Parsing with SheetJS

The foundational challenge for Cartulary is the reliable parsing of diverse archival spreadsheets within the client-side browser environment. The goal is to transform a raw `.xlsx` or `.csv` file into a structured, usable format, primarily an array of objects, that serves as the basis for all subsequent processing. This process involves navigating browser-specific APIs, addressing complex character encoding issues, handling non-standard spreadsheet features like merged cells, and managing performance with potentially large datasets. The provided materials indicate that SheetJS (js-xlsx) is a well-suited library for this task due to its broad format support and dual-browser/Node.js compatibility [[6](https://www.npmjs.com/package/xlsx), [88](https://cloud.tencent.com/developer/article/2164147)]. Its Community Edition provides battle-tested solutions for extracting data from complex spreadsheets, making it a pragmatic choice for this project [[6](https://www.npmjs.com/package/xlsx)].

The recommended implementation begins with handling the file input event from the browser. Whether through a standard `<input type="file">` element or a drag-and-drop interface, the result is a `File` object. The correct and most robust sequence of operations starts with using the `FileReader` API to read this file [[5](https://juejin.cn/post/6997361645181730852)]. The critical insight derived from the analysis is to avoid using `readAsText()` initially, as it forces the browser to interpret the byte stream using a single encoding (typically UTF-8), which often fails with legacy archival files containing characters from other encodings like Windows-1252 or various ISO 8859 variants [[4](https://stackoverflow.com/questions/64284226/special-character-issue-in-excel-for-csv-file-sheetjs-prepend-bom-for-utf8-csv), [82](https://www.volcengine.com/article/165914)]. Instead, the file should be read into an `ArrayBuffer` using `reader.readAsArrayBuffer(file)` [[73](https://www.volcengine.com/article/888853)]. This preserves the raw binary data intact. This buffer is then passed to SheetJS, which has sophisticated internal logic to deduce the file type and begin parsing. SheetJS can then correctly interpret the byte stream based on the file's internal headers and specified options. Once the workbook is loaded, the first worksheet is typically selected, and the `sheet_to_json` function is used to convert the worksheet data into a JavaScript array of objects. To ensure the first row of the spreadsheet is treated as the header keys for the resulting objects, the `{ header: 1 }` option must be passed to `sheet_to_json` [[50](https://stackoverflow.com/questions/52215676/sheetjs-json-to-sheet-renaming-headers)]. A significant pitfall in archival data processing is character encoding inconsistency. Legacy systems and software have produced files with a hodgepodge of encodings, and simply reading them as text will lead to garbled special characters, non-Latin scripts, or accented characters being corrupted [[4](https://stackoverflow.com/questions/64284226/special-character-issue-in-excel-for-csv-file-sheetjs-prepend-bom-for-utf8-csv)]. While SheetJS itself uses the `utf8` encoding by default, it also offers a powerful `codepage` option to handle files in other encodings [[109](https://zhuanlan.zhihu.com/p/138400272)]. This option allows the developer to specify the code page of the source file, forcing SheetJS to interpret the byte stream correctly. For instance, when dealing with Arabic data, one might use `codepage: 28596` for ISO 8859-6 [[111](https://stackoverflow.com/questions/56966949/how-can-i-use-sheetjs-to-read-arabic-data-from-excel-file)]. While no single setting works for all scenarios, the recommended strategy is to first attempt parsing with default settings. If character corruption is detected, a fallback mechanism could try common code pages. Furthermore, SheetJS handles Byte Order Marks (BOMs), which are signatures at the beginning of a text file indicating its encoding (e.g., `ï»¿` for UTF-8 BOM). Reading as an ArrayBuffer helps mitigate issues related to BOMs that can trip up simpler parsers that rely solely on text-based reading [[5](https://juejin.cn/post/6997361645181730852), [44](https://community.make.com/t/encoding-csv-file-to-utf-8-with-bom/95265), [82](https://www.volcengine.com/article/165914)].

Archivists frequently use merged cells in spreadsheets to create a visual hierarchy, but this is purely a formatting feature and not a structural indicator of parent-child relationships [[22](https://www.researchgate.net/publication/301931689_Using_and_Exploring_Hierarchical_Data_in_Spreadsheets)]. SheetJS exposes metadata about merged cell ranges through the `!merges` property on a worksheet object [[7](https://stackoverflow.com/questions/53516403/sheetjs-xlsx-how-to-write-merged-cells), [42](https://techcommunity.microsoft.com/discussions/excelgeneral/how-can-i-find-merged-cells-in-a-large-spreadsheet/4516238)]. A safe and robust strategy is to treat merged cells as presentation artifacts only. Upon detecting a merged region, the application should programmatically "unmerge" the data by filling the empty cells within the merge range with the value from the top-left cell of that range. This preserves the intended data value without introducing spurious parent-child links that could corrupt the logical tree structure. It is important to note that merged cells are an Excel-specific trick; internally, Excel redirects calls to the parent cell, making them difficult to work with directly and reinforcing the wisdom of flattening their content [[112](https://forum.uipath.com/t/can-t-read-merged-cells/13754)].

Finally, performance with large datasets is a valid concern. Spreadsheets with hundreds or thousands of rows and many columns can cause performance bottlenecks or even out-of-memory errors if parsed synchronously on the main thread [[8](https://stackoverflow.com/questions/51547676/reading-large-excel-files-with-sheetjs-js-xlsx)]. To mitigate this, several strategies can be employed. First, providing a preview of the first few rows of the spreadsheet in the UI immediately after upload gives the user feedback and allows them to abort the process if the file appears to be too large before committing resources [[81](https://stackoverflow.com/questions/52715836/sheetjs-shows-blank-rows)]. For very large files, offloading the parsing operation to a Web Worker is a more advanced optimization to prevent blocking the UI thread, although this adds complexity [[9](https://stackoverflow.com/questions/50569062/validate-huge-xml-in-browser-side-using-js)]. Using the `readAsArrayBuffer` method is generally more efficient for large files than `readAsText` because it avoids premature string conversion and encoding interpretation [[73](https://www.volcengine.com/article/888853)].

## Namespace-Aware EAD3 XML Generation with xmlbuilder2

Once the spreadsheet data has been parsed and structured, the next core task is generating a valid EAD3 XML document. This requires precise control over XML namespaces, declaration of schema locations, inclusion of processing instructions, and producing human-readable output. The research indicates that `xmlbuilder2` is an excellent tool for this purpose, offering a fluent, chainable API that simplifies the construction of complex XML trees [[78](https://www.npmjs.com/package/xmlbuilder2?activeTab=code)]. Its design as a wrapper around DOM nodes makes it intuitive to build documents programmatically while ensuring syntactic correctness [[78](https://www.npmjs.com/package/xmlbuilder2?activeTab=code)].

To create the root `<ead>` element with the required EAD3 namespaces, a specific code pattern must be followed. The default namespace (`xmlns="http://ead3.archivists.org/schema/"`) and the `xsi` namespace (`xmlns:xsi="http://www.w3.org/2001/XMLSchema-instance"`) must be declared on the root element. The `xsi:schemaLocation` attribute then ties the EAD3 namespace to its corresponding XSD file. `xmlbuilder2`'s `.att()` method is used to add these attributes. A minimal working code pattern would look as follows:

```javascript
const builder = require('xmlbuilder2');
const ead3NS = 'http://ead3.archivists.org/schema/';
const xsiNS = 'http://www.w3.org/2001/XMLSchema-instance';

// Create the document with XML declaration
const doc = builder.create('ead', { 
  version: '1.0', 
  encoding: 'UTF-8' 
})
// Add namespace declarations
.att('xmlns', ead3NS)
.att('xmlns:xsi', xsiNS)
// Add the schemaLocation attribute
.att(xsiNS + ':schemaLocation', ead3NS + ' http://www.archivists.org/ead/ead.xsd') // Note: URL may need updating

// At this point, you would append child elements like <eadheader> and <archdesc>
// ...

// End the document and serialize it to a string with pretty-printing enabled
const eadXmlString = doc.end({ prettyPrint: true });
```

This pattern demonstrates several key points. First, the `version` and `encoding` properties in the `create()` call automatically generate the `<?xml version="1.0" encoding="UTF-8"?>` processing instruction, which is essential for any XML document [[46](https://www.ibm.com/docs/en/app-connect/11.0.0?topic=body-xml-processinginstruction), [47](https://bugzilla.mozilla.org/show_bug.cgi?id=175946)]. Second, adding an attribute named `'xmlns'` creates the default namespace, while adding `'xmlns:xsi'` creates the prefix for the `xsi` namespace used in the `schemaLocation` attribute [[76](https://stackoverflow.com/questions/62884471/xml-namespace-issues-when-building-an-xml-file-in-nodejs-xmlbuilder2)]. Third, serialization is handled by the `.end()` method, which accepts an options object. Setting `{ prettyPrint: true }` ensures the final output is indented and formatted for human readability, a crucial feature for archivists who will need to review the generated XML [[54](https://stackoverflow.com/questions/5657181/use-xslt-to-pretty-print-xml-xhtml-without-corrupting-namespace-info)]. There are no widely reported issues with `xmlbuilder2` producing spurious `xmlns=""` declarations on child elements in modern usage, as its internal handling of namespace inheritance is generally robust [[78](https://www.npmjs.com/package/xmlbuilder2?activeTab=code)].

While `xmlbuilder2` is a strong choice, it is worth considering alternatives. The `fast-xml-parser` library is noted for being significantly faster than other pure-JavaScript XML parsers [[16](https://www.npmjs.com/package/fast-xml-parser)]. However, its primary strength lies in *parsing* XML into JavaScript objects, not in *building* them. For a use case like Cartulary, where the goal is to generate a structured, readable XML document from data, the object-oriented, chainable syntax of `xmlbuilder2` is arguably more aligned with the development workflow and produces more predictable output [[78](https://www.npmjs.com/package/xmlbuilder2?activeTab=code)]. Given that the generation speed is unlikely to be a bottleneck compared to the parsing of large spreadsheets, the benefits of `xmlbuilder2`'s clarity and ease of use outweigh the marginal performance gains of a parser-focused alternative. Therefore, `xmlbuilder2` remains the recommended library for this project.

## Hierarchy Tree Construction from Flat Tabular Data

A central and complex part of Cartulary is transforming flat, tabular data into a hierarchical tree structure, mirroring the nested nature of archival descriptions in EAD3. The user's requirement specifies three distinct methods archivists use to structure data in spreadsheets: a level column, dotted component IDs, and an explicit parent_id reference. A sound architectural decision is to implement separate, robust parsing logic for each method, with each converging on a single, normalized intermediate data structure. This modular approach simplifies testing, debugging, and maintenance, which aligns with the project's goal of long-term viability [[41](https://dev.to/avithe1/deploy-vite-react-project-with-react-router-without-404-on-github-pages-h6p), [61](https://dev.to/lico/handling-404-error-in-spa-deployed-on-github-pages-246p)]. The ideal converged format is an adjacency list, represented as a flat array of node objects. Each object in this array contains its unique ID, its parent's ID, and all the data associated with that row from the spreadsheet. This structure is highly versatile for subsequent validation and serialization tasks [[79](https://stackoverflow.com/questions/21342596/tree-structure-from-adjacency-list)].

For the **Level Column** method, where hierarchy is inferred from a sequence of values (e.g., collection, series, file), the algorithm relies on a stack-based approach. As we iterate through the rows, we maintain a stack of the currently active parent nodes. For each new row, we determine its target depth based on its level value. We pop nodes from the stack until its length is one less than the target depth. The new row's parent is the last node remaining on the stack. We then push the new row onto the stack. This gracefully handles sequences where levels increment by one. Edge cases, such as a level jumping back up by more than one, would require popping multiple nodes, while a level staying the same would involve popping and replacing the last node on the stack. A level going backward would require clearing the stack to the appropriate depth.

For the **Dotted Component ID** method, the logic is more direct but requires careful validation. Each row with an ID like `1.1.2` has a parent whose ID is the same string with the last segment removed, in this case `1.1`. If the ID contains no dots, the node is a root-level component with a `parentId` of `null`. The primary edge cases here are malformed IDs (e.g., `1..2`, `1.a.2`) and duplicate IDs. The algorithm must first validate the ID format using a regular expression. After normalizing the ID to find the parent ID, it must check if a node with that parent ID exists in the dataset before establishing the relationship to avoid creating orphans.

For the **Explicit `parent_id`** method, the logic involves resolving references between rows. A simple iteration and search would be inefficient. A more robust approach is a two-pass algorithm. In the first pass, iterate through all rows, create a node object for each, and store them in a Map or plain object indexed by their own `unitid`. In the second pass, iterate through the rows again. For each row, look up its parent in the map using its `parent_id` value. If found, establish the parent-child link; if not found, flag it as an orphan. This efficiently resolves all parent references in $O(n)$ time.

Once the adjacency list is constructed from any of the three methods, a unified set of validation checks can be performed. Orphaned rows are detected by checking if a node's `parentId` corresponds to an existing node ID. Duplicate `<unitid>` values are found by iterating through the list and using a `Set` to track seen IDs. Circular references are a classic graph theory problem and must be checked for every node in the tree. A robust algorithm for this is an iterative Depth-First Search (DFS) [[52](https://www.geeksforgeeks.org/dsa/detect-cycle-in-a-graph/), [53](https://www.w3schools.com/dsa/dsa_algo_graphs_cycledetection.php)]. For each unvisited node, we start a DFS, maintaining a "recursion stack" (a Set or List) of nodes in the current path. If we encounter a node that is already in this stack, we have found a cycle. We must repeat this process for every node in the graph to ensure no cycles exist anywhere [[35](https://stackoverflow.com/questions/1347533/how-do-i-detect-circular-logic-or-recursion-in-a-multi-levels-references-and-dep), [89](https://stackoverflow.com/questions/44024423/how-to-detect-cycles-in-directed-graph-in-iterative-dfs), [95](https://dev.to/rohithv07/graph-algorithm-cycle-detection-in-directed-graph-using-dfs-4bl5)]. The standard for serializing this tree into EAD3 is a depth-first pre-order traversal, where a node is processed before its children [[25](https://dev.to/christiankastner/studying-graphs-pt-2-breadth-and-depth-first-search-3a72), [70](https://www.linkedin.com/posts/shashidhar-reddy-erri-b94b0898_softwareengineering-algorithms-datastructures-activity-7427913162231906305-Jx7P)]. This naturally creates the nested `<c>` elements required by the EAD3 schema. An iterative DFS using a stack is preferred over recursion to prevent stack overflow errors with deeply nested hierarchies [[69](https://www.volcengine.com/article/198856), [71](https://dev.to/shahrouzlogs/day-64-python-depth-first-search-dfs-on-tree-stack-based-iterative-traversal-for-deep-4gej)].

## In-Browser Validation Strategy

Providing immediate and actionable feedback to the user is paramount for a tool like Cartulary. The question of whether to implement full XSD validation versus a practical rule-based validator is central to the project's scope and user experience. While the capability to perform full XSD validation in the browser exists through WebAssembly (WASM) libraries like `libxmljs-wasm` [[100](https://www.npmjs.com/package/libxml2-wasm)], attempting this path introduces significant drawbacks that make it a higher-risk endeavor for this specific application [[39](https://dev.to/zny10289/webassembly-in-2026-the-quiet-revolution-that-finally-delivered-8ff)]. The primary concerns are bundle size, performance overhead, and maintenance complexity. A WASM module for a full XML parser and validator can add substantial weight to the application, potentially exceeding desirable limits for a fast-loading client-side tool [[43](https://stackoverflow.com/questions/57542777/reducing-bundle-size-of-a-production-application), [75](https://community.cloudflare.com/t/build-failure-due-to-pages-functions-bundle-size-exceeding-limit/733027)]. Even on modern hardware, the computational cost of validating against a comprehensive schema like EAD3 can slow down the browser, especially on lower-end devices. Furthermore, relying on a WASM dependency adds another layer of complexity for updates and bug fixes. For a tool aimed at helping archivists streamline their workflow, prioritizing speed and usability is more important than achieving technical perfection in validation. Therefore, a curated, rule-based validator represents a far more practical and valuable investment of effort.

A well-designed rule-based validator focuses on catching the most frequent and impactful errors that archivists commonly make during manual data entry. The user's suggested starting list of checks covers the majority of these high-value issues. The implementation should prioritize rules that are computationally inexpensive yet highly effective. These rules include:

1.  **Missing Required Fields:** Based on EAD3 standards, certain elements like `<unitid>` and `<unittitle>` are mandatory for every `<c>` component. The validator should verify the presence of these fields for each node in the hierarchy.
2.  **Orphaned Components:** As established in the hierarchy construction phase, this check identifies any node whose `parentId` does not correspond to the `id` of a valid, existing parent node.
3.  **Duplicate `<unitid>` Values:** A globally unique identifier is a cornerstone of EAD. This check iterates through the adjacency list and flags any duplicate `unitid` values, which would break the document's integrity.
4.  **Malformed Date Strings:** While full compliance with the `xsd:date` datatype is complex, catching obvious errors is sufficient. A simple regex can validate dates against common patterns like `YYYY-MM-DD` or basic ranges (`1900/jan/1950`). Full compliance with datatypes is defined in specifications like XML Schema Part 2 [[99](https://www.w3.org/TR/xmlschema-2/)].
5.  **Circular Parent References:** This check, implemented using an iterative DFS cycle detection algorithm [[52](https://www.geeksforgeeks.org/dsa/detect-cycle-in-a-graph/), [95](https://dev.to/rohithv07/graph-algorithm-cycle-detection-in-directed-graph-using-dfs-4bl5)], ensures the hierarchy remains a valid tree structure without loops.

The user experience (UX) of error reporting is as important as the validation logic itself. Errors should not be presented as a cryptic list of XML lines. Instead, they must be mapped back to the context of the original spreadsheet. When a validation error is detected—for example, an orphaned row—the system should store the index of the problematic row from the parsed JSON array. When displaying the error message (e.g., "Component 'COLL1_SUB2' is orphaned"), the UI should highlight the corresponding row in the column-mapping interface. This "row highlighting" pattern allows the archivist to instantly see the faulty data and its surrounding context, enabling them to fix the issue without needing to re-map columns or cross-reference documents manually. Open-source examples in similar tools often employ this technique to improve data correction workflows [[22](https://www.researchgate.net/publication/301931689_Using_and_Exploring_Hierarchical_Data_in_Spreadsheets), [62](https://stackoverflow.com/questions/29102610/jquery-parse-dom-and-create-a-tree-structure-composite-pattern)]. By combining a focused set of rules with intuitive, row-referenced feedback, Cartulary can build trust with its users by providing tangible value and improving their overall productivity.

## Deployment and Maintainability on GitHub Pages

Deploying a modern web application to GitHub Pages requires careful consideration of the build process, dependency management, and deployment configuration. While it is technically possible to create a "vanilla JS" application that loads all dependencies via CDNs, this approach quickly becomes unmanageable for a project with non-trivial logic and multiple external libraries like SheetJS and xmlbuilder2 [[28](https://www.volcengine.com/article/169760)]. Managing script load order, avoiding global namespace conflicts, and handling versioning become significant maintenance burdens. For a project aiming for low long-term maintenance, adopting a lightweight bundler is strongly recommended.

Vite is the recommended baseline for this project. It offers an exceptional balance of simplicity and power for modern JavaScript development. Built around native ES modules, Vite enables tree-shaking, a crucial optimization that removes unused exports from libraries in the final production bundle [[57](https://dev.to/morewings/how-to-build-a-tree-shakable-library-with-vite-and-rollup-16cb)]. This means that if your application only uses a small portion of SheetJS or xmlbuilder2, the dead code will be eliminated, resulting in a smaller download size for the end-user [[92](https://dev.to/pouja/everything-about-esm-and-treeshaking-5f4l)]. Vite also provides a fast, hot-reloading development server that significantly improves developer productivity. Its configuration is minimal, and it integrates seamlessly with modern JavaScript and TypeScript.

Deploying a Vite-built Single Page Application (SPA) to GitHub Pages is a well-documented process, though it comes with one key challenge: handling client-side routing and the 404 error [[60](https://stackoverflow.com/questions/76115927/page-not-found-react-vite-app-not-routing-correctly-on-github-pages)]. By default, when a user navigates directly to a deep link (e.g., `/my-page`) or refreshes the page, the server doesn't know how to handle it, resulting in a 404 error. GitHub Pages, by design, serves the `index.html` file for the root path. To solve this for SPAs, a common solution is to create a `404.html` file in the root directory of the `dist` output. This file should contain a simple HTML snippet that redirects all non-root requests back to `index.html`. This tricks the browser into letting the client-side router handle the URL, mimicking the behavior of a traditional multi-page site [[61](https://dev.to/lico/handling-404-error-in-spa-deployed-on-github-pages-246p)]. Vite's official documentation provides clear guidance on configuring the `base` and `build.outDir` options to prepare the application for deployment to a subdirectory on GitHub Pages (e.g., `https://username.github.io/project-name/`) [[41](https://dev.to/avithe1/deploy-vite-react-project-with-react-router-without-404-on-github-pages-h6p)]. Attempting to deploy a complex SPA without a bundler would force manual management of these details, increasing the risk of errors and making future updates more difficult. The minor setup cost of Vite is vastly outweighed by the long-term gains in maintainability, performance, and developer experience.

## Synthesis and Final Recommendations

The proposed technical stack for Cartulary—SheetJS for parsing, xmlbuilder2 for XML generation, and a lightweight bundler like Vite for maintainability—is not only viable but represents a robust and pragmatic choice for the stated goals. The analysis confirms that each component is well-suited to its respective task within the constraints of a client-side-only browser application deployed on GitHub Pages. The project's success hinges on implementing the four core challenges with attention to detail, particularly in handling edge cases and optimizing the user experience.

The implementation guidance synthesized from the provided sources leads to a clear and actionable path forward. For spreadsheet parsing, the recommended workflow involves using the `FileReader` API to read files into an `ArrayBuffer`, which is then fed to SheetJS. This approach circumvents many common character encoding issues that plague legacy archival files, though developers should remain prepared to use SheetJS's `codepage` option as a fallback [[109](https://zhuanlan.zhihu.com/p/138400272)]. Visually hierarchical merged cells should be programmatically flattened rather than interpreted as structural data to preserve data integrity [[112](https://forum.uipath.com/t/can-t-read-merged-cells/13754)]. For XML generation, `xmlbuilder2` provides an elegant and effective API for constructing namespace-aware EAD3 documents with proper processing instructions and pretty-printing, fulfilling the core requirements of the task [[54](https://stackoverflow.com/questions/5657181/use-xslt-to-pretty-print-xml-xhtml-without-corrupting-namespace-info), [78](https://www.npmjs.com/package/xmlbuilder2?activeTab=code)].

The architecture for hierarchy construction, which employs separate parsing logic for the three distinct structuring methods (level column, dotted IDs, `parent_id`), is a sound design decision. Converging all methods to a unified adjacency list representation simplifies subsequent validation and serialization processes [[79](https://stackoverflow.com/questions/21342596/tree-structure-from-adjacency-list)]. This shared format is perfectly suited for the iterative depth-first search algorithms needed to validate for orphans, duplicates, and circular references—a critical step for ensuring the structural integrity of the final EAD3 document [[52](https://www.geeksforgeeks.org/dsa/detect-cycle-in-a-graph/), [95](https://dev.to/rohithv07/graph-algorithm-cycle-detection-in-directed-graph-using-dfs-4bl5)]. Prioritizing a focused, rule-based validator over a full XSD validation is the most strategic choice. It delivers the highest value by catching the most common and impactful errors that archivists make, while avoiding the significant bundle size and performance penalties associated with WASM-based XSD validators [[39](https://dev.to/zny10289/webassembly-in-2026-the-quiet-revolution-that-finally-delivered-8ff), [75](https://community.cloudflare.com/t/build-failure-due-to-pages-functions-bundle-size-exceeding-limit/733027)]. Presenting these validation errors with direct row references back to the original spreadsheet is essential for creating a tool that archivists can trust and use effectively [[22](https://www.researchgate.net/publication/301931689_Using_and_Exploring_Hierarchical_Data_in_Spreadsheets)].

Finally, deploying the application using Vite is strongly advised. It addresses the long-term maintenance burden that a pure vanilla JS/CDN approach would introduce, leveraging modern tooling like ES modules and tree-shaking to produce a lean, performant, and manageable application [[57](https://dev.to/morewings/how-to-build-a-tree-shakable-library-with-vite-and-rollup-16cb), [92](https://dev.to/pouja/everything-about-esm-and-treeshaking-5f4l)]. While the initial setup requires more configuration than a simple HTML file, the benefits in developer productivity and application quality are substantial. By following these recommendations, the Cartulary project is well-positioned to succeed as a reliable, efficient, and user-friendly tool for transforming archival data from spreadsheets into properly formatted EAD3 XML.
