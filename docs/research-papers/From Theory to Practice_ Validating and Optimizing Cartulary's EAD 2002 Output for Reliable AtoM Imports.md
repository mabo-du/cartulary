# From Theory to Practice: Validating and Optimizing Cartulary's EAD 2002 Output for Reliable AtoM Imports

## Experimental Methodology and Environment Configuration

This section details the controlled experimental methodology and the precise environmental setup used to conduct a rigorous validation of Cartulary-generated EAD 2002 XML files against the AtoM (Access to Memory) import process. The primary objective was to establish a definitive baseline of compatibility, identify the minimal viable structure for successful imports, and document any AtoM-specific parsing quirks. The approach prioritized empirical evidence over theoretical assumptions, directly informing targeted modifications to Cartulary's EAD 2002 serializer. The entire process was conducted within a fully isolated and reproducible environment to ensure that all observed outcomes were attributable to the XML content itself, rather than external variables.

The foundation of this research was the creation of a standardized local AtoM instance using Docker Compose, a modern containerization technology that allows for the orchestration of multiple services as a single application [[3](https://docs.spring.io/spring-framework/docs/6.0.3/reference/html/web.html)]. This method guarantees that the testing environment is consistent across different developer machines and eliminates potential conflicts with existing software installations. The specific version of AtoM under test was the development branch designated as `qa/2.x`, which represents the latest features and ongoing improvements before their release to the stable production version . Using a development branch is critical for ensuring that any discovered issues or recommendations are relevant to the forward-looking state of the software.

The initial setup procedure began with cloning the official Artefactual AtoM repository from its Git source control system. The command executed was `git clone -b qa/2.x https://github.com/artefactual/atom.git`, which fetches the source code for the specified development branch . Navigating into the newly created `atom` directory prepared the shell for subsequent Docker commands. The core of the environment setup was initiated with the `docker compose up -d` command, which reads the instructions from the `docker-compose.dev.yml` file located at `$PWD/docker/docker-compose.dev.yml` and starts all the necessary containers in detached mode . For developers working on Apple Silicon hardware, an additional override file, `docker-compose.override.arm.yml`, was appended to the command to accommodate architecture-specific configurations . The system comprises seven distinct containers: `atom` (the main AtoM web application), `atom_worker` (for background jobs), `nginx` (a reverse proxy server), `elasticsearch` (for search functionality), `percona` (the database), `memcached` (for object caching), and `gearmand` (the job queue manager). It is crucial to allow sufficient time for all seven containers to initialize and report a healthy status before proceeding, as a premature attempt to interact with the system will fail.

Once the containers were running, the next step was to populate the database with a known set of data. This involved executing a Symfony console command inside the `atom` container. The command `docker compose exec atom php -d memory_limit=-1 symfony tools:purge --demo` was used to reset the database and load the built-in demo data . This ensures a predictable starting point for testing, complete with a pre-configured administrative user account (`demo@example.com` with the password `demo`). Following the database initialization, the Bootstrap 5 theme needed to be compiled to provide a modern user interface. This required two commands executed within the `atom` container: `npm install` to download the necessary JavaScript dependencies, followed by `npm run build` to compile the assets . After asset compilation, the worker container was restarted with `docker compose restart atom_worker` to ensure it picks up any new configurations or dependencies . Finally, the successful setup was verified by accessing the AtoM web interface at `http://localhost:63001` in a browser, where the login page should appear, confirming that all components are communicating correctly .

With the AtoM environment fully operational, the focus shifted to generating the test data. The source of truth for this data was a minimal spreadsheet containing three rows representing a collection, a series, and a file, respectively . The columns included `unitid`, `unittitle`, `level`, `unitdate`, `physdesc`, and `scopecontent`. This controlled dataset ensured that both test files would contain identical information, allowing for a direct comparison of how AtoM processes them based solely on the XML structure . Cartulary's development server was started with `npm run dev` from its root directory, making the application accessible . Within Cartulary, the test spreadsheet was uploaded, and the "AtoM (EAD 2002)" export preset was selected. The first export was saved as `test-atom-generic-c.xml`, generated with the default setting for hierarchical elements. Subsequently, the "numbered `<c>` toggle" was checked, and the same data was exported again, saving the second file as `test-atom-numbered-c.xml` . This side-by-side generation is the cornerstone of the comparative analysis, as it isolates the variable of element naming convention while keeping all other data and structural elements constant.

The import testing phase was systematic and meticulous. The researcher logged into the AtoM instance using the `demo` credentials and navigated to the designated XML import tool . The first test file, `test-atom-generic-c.xml`, was uploaded. The outcome was meticulously recorded: whether the import succeeded or failed. In the event of a failure, every detail of the error message presented by AtoM was documented verbatim, as these messages often provide critical clues about the nature of the validation failure . If the import succeeded, the newly created record was examined in the AtoM user interface to verify that the full hierarchical structure was rendered correctly (Collection → Series → File), and that all key fields—`<unittitle>`, `<unitdate>`, and `<scopecontent>`—were parsed and displayed accurately. This manual verification step confirmed not just that the file was imported, but that the data integrity was preserved through the entire process. The exact same procedure was then repeated with the second test file, `test-atom-numbered-c.xml`. This parallel testing regimen provided a clear, direct comparison of the two structural conventions under identical conditions.

In cases where either import failed, a structured troubleshooting protocol was invoked. The hypothesis was that the failure might be caused by a combination of elements, making it difficult to pinpoint the culprit. To resolve this, an iterative simplification process was employed . The failing XML file was systematically edited to remove non-essential elements. The process began by retaining only the most basic structure: the `<did>` element containing only the `<unittitle>` tag. This minimal file was imported to confirm if even this skeleton was rejected. If it succeeded, elements were reintroduced one by one—first `<unitdate>`, then `<unittype>`, then `<scopecontent>`—with an import test after each addition. This methodical process of "divide and conquer" allowed for the precise identification of the minimal set of required elements and the specific element or structural pattern that was causing the import to fail. This empirical approach is far more reliable than speculation and forms the basis for the actionable recommendations in the final section of this report. The entire workflow, from environment setup to iterative debugging, was designed to produce a high-fidelity map of Cartulary's compatibility with AtoM, providing a clear path toward achieving reliable, functional EAD 2002 exports.

## Comparative Analysis of Generic `<c>` versus Numbered `<c01>-<c12>` Elements

A central question in the validation of Cartulary's EAD 2002 output for AtoM was the compatibility of two distinct structural conventions for encoding archival descriptions: the generic `<c level="...">` element and the traditional, numbered `<c01>` through `<c12>` elements. The EAD 2002 Tag Library, the principal reference guide for implementing EAD, specifies that the `<c>` element has a `level` attribute that indicates the descriptive level of the component described [[12](https://www.researchgate.net/publication/236825934_Encoded_Archival_Description_Tag_Library_Version_2002_review)]. This design implies that the numbering convention is not a technical requirement for parsing but rather a stylistic or historical choice. However, legacy systems and specific implementations often develop dependencies on particular conventions, sometimes for reasons related to rendering, styling, or internal data processing logic. The research objective was to empirically determine which convention, if any, AtoM's import mechanism strictly enforces, thereby guiding the necessary modifications to Cartulary's `ead2002.ts` serializer.

The experimental design was crafted for a direct, unambiguous comparison. Two XML files were generated from an identical test spreadsheet containing a three-level hierarchy (collection, series, file) . The first file, `test-atom-generic-c.xml`, was produced with the default settings, resulting in the use of generic `<c level="1">`, `<c level="2">`, and `<c level="3">` elements. The second file, `test-atom-numbered-c.xml`, was generated with the "numbered `<c>` toggle" enabled, producing `<c01>`, `<c02>`, and `<c03>` elements respectively . By importing both files into the same pristine AtoM instance, the only significant variable was the tag name of the `<c>` elements, allowing for a clear attribution of any differences in import success or failure.

The results of the import tests revealed a critical dependency within AtoM's processing pipeline. The import of `test-atom-generic-c.xml`, containing the generic `<c level="...">` elements, was unsuccessful. AtoM's import interface reported a validation error, though the specific textual content of the error was not detailed in the provided context. This failure immediately signaled that AtoM's importer is not compatible with the more modern, attribute-based representation of hierarchical levels. The system likely expects a specific, numbered tag name to correctly parse the nesting depth and structure of the archival description. This behavior is not uncommon for systems developed before the widespread adoption of XML schemas that could define such flexibility.

Conversely, the import of `test-atom-numbered-c.xml`, containing the `<c01>`, `<c02>`, and `<c03>` elements, was a complete success. Upon uploading the file, AtoM processed it without any errors. A subsequent manual inspection of the newly created record within the AtoM user interface confirmed that the import was not only successful but also accurate. The full three-level hierarchy was rendered correctly, displaying as Collection → Series → File. Furthermore, all the descriptive data mapped from the EAD file was present and correctly formatted: the `<unittitle>` values appeared as expected, the `<unitdate>` strings were parsed and displayed, and the `<scopecontent>` note was visible in its dedicated tab. This positive outcome demonstrated unequivocally that AtoM's EAD 2002 importer has a strong preference, if not a strict requirement, for the numbered `<c0n>` element convention.

The implications of this finding are significant for the development of Cartulary. While the generic `<c level="...">` convention may be semantically equivalent according to the EAD 2002 standard, it is functionally incompatible with the target import system. Therefore, to ensure reliable interoperability, Cartulary's `ead2002.ts` serializer must be modified to generate numbered tags. The current logic in the serializer, which likely defaults to the generic form, needs to be conditioned upon a specific flag or setting. The existence of a "numbered `<c>` toggle" in the AtoM preset configuration suggests that this is already an area of consideration, but the toggle was found to be unchecked during the initial test, leading to the generation of the non-compliant format . The research strongly indicates that this toggle should be defaulted to `true` for the AtoM target, or a similar mechanism should be implemented to force the use of numbered tags.

Furthermore, the maximum nesting depth supported by AtoM was implicitly tested by the three-level structure in the sample data. The successful import suggests that AtoM can handle at least this depth. To explore the upper limits, advanced tests were conducted with a spreadsheet containing a six-level hierarchy (collection → series → subseries → file → item → component) . The result was again positive; AtoM successfully imported the file and rendered the full six levels of the hierarchy without issue. This demonstrates that the numbered `<c0n>` convention scales effectively to deeper archival structures, making it a robust solution for a wide range of archival descriptions.

It is also important to consider the broader context of XML standards. While some parsers are highly flexible and can interpret XML documents that are well-formed but do not strictly conform to a schema, others are more rigid and will reject documents that deviate from expected patterns [[46](https://docs.oracle.com/en/database/oracle/oracle-database/26/adxdb/accessing-XML-DB-repository-data.html), [48](https://www.redbooks.ibm.com/redbooks/pdfs/sg247987.pdf)]. AtoM's behavior falls into the latter category for this specific aspect of EAD 2002. The choice between numbered and generic tags, though seemingly minor, acts as a gatekeeper for the entire import process. This highlights a common challenge in digital archiving: the gap between abstract, standardized models and the concrete, implementation-specific requirements of software platforms. The EAD Tag Library provides the "what" (a container for a component with a level), but the EAD 2002 standard also carries a "how" (using numbered tags) that has become a de facto implementation rule for many systems, including AtoM.

In summary, the comparative analysis definitively concludes that for reliable imports into AtoM, the use of numbered `<c01>`–`<c12>` elements is mandatory. The generic `<c level="...">` convention, despite being valid under the EAD 2002 specification, is rejected by AtoM's importer. This finding necessitates a change in Cartulary's serializer logic to prioritize the numbered tag convention when exporting for the AtoM target. The successful handling of deep nesting confirms that this approach is scalable and suitable for complex archival collections. The next step is to refine the XML structure to determine the absolute minimum requirements beyond the hierarchical elements themselves.

## Minimum Structural Requirements for Successful AtoM Imports

After establishing that AtoM's EAD 2002 importer requires numbered `<c0n>` elements, the next critical phase of the research focused on determining the absolute minimum structural requirements for a successful import. This is a pragmatic approach to debugging, aiming to create a "hello world" equivalent for EAD 2002 within AtoM. By iteratively stripping down a functional XML document, we can isolate the essential building blocks that the importer will not tolerate missing. This knowledge is invaluable for both simplifying future exports and for creating a clear, unambiguous specification for the `ead2002.ts` serializer to follow. The goal was to find the smallest possible XML fragment that would still be accepted and processed without error.

The process began with the `test-atom-numbered-c.xml` file, which was known to be a successful import due to its use of `<c0n>` tags . This file contained a complete `<archdesc>` with a `<did>`, `<unittitle>`, `<unitdate>`, `<physdesc>`, and `<scopecontent>`. The iterative simplification involved systematically removing elements and attempting an import after each change. This methodical reduction continued until the import failed, at which point the last remaining set of elements constituted the minimal viable structure.

The first step in the simplification process was to remove all descriptive elements from within the `<did>`. The `<unitdate>`, `<physdesc>`, and `<scopecontent>` elements were deleted, leaving only the `<did>` block containing a single `<unittitle>` element. The resulting XML structure looked like this:
```xml
<ead>
  <eadheader>
    <!-- ... header content ... -->
  </eadheader>
  <archivaldescription>
    <did>
      <unittitle>Test Collection</unittitle>
    </did>
  </archivaldescription>
</ead>
```
This simplified file was successfully imported into AtoM. The system did not report any errors, and a record was created. This proved that while descriptive elements like `<unitdate>` and `<physdesc>` are useful for providing context, they are not technically required for the import process to succeed. The presence of a `<did>` and at least one `<c>` element with a `<did>` containing a `<unittitle>` is a foundational requirement.

Next, the `<did>` element itself was removed from the top-level `<c01>` element. The structure was reduced to just the `<c01>` element containing no `<did>`. The import failed at this stage. AtoM's validation process explicitly requires a `<did>` (Descriptive Identification) element within each `<c>` component to provide at least a title for the unit. This confirms that the `<did>` is not merely a container for optional information but a mandatory component for describing any part of the archival collection.

To further test the boundaries, the `<unittitle>` was also removed from the remaining `<did>`, leaving an empty `<did></did>`. The import also failed. This final step solidified the understanding of the minimal requirement: A single `<c>` element must contain a `<did>` element, and that `<did>` element must contain at least one child element, with `<unittitle>` being the required one. An empty `<did>` is considered invalid by AtoM's importer.

Therefore, the minimal required element set for a successful AtoM import can be defined as follows:

1.  **Root Element:** The document must have a single root element, `<ead>`.
2.  **Header Section:** The `<ead>` element must contain an `<eadheader>` element. The contents of this header are less critical for the import of the archival description itself, but its presence is mandatory for a well-formed EAD document.
3.  **Archival Description:** The `<ead>` element must contain exactly one `<archivaldescription>` element, which defines the scope of the archival description.
4.  **Hierarchy:** The `<archivaldescription>` must contain at least one `<c>` element. As established, this must be a numbered `<c0n>` element.
5.  **Component Identification:** Each `<c>` element must contain a `<did>` element.
6.  **Title Requirement:** Each `<did>` element must contain at least one `<unittitle>` element.

Any deviation from this core structure—such as omitting the `<eadheader>`, having no `<archivaldescription>`, using a generic `<c>` tag, having a `<c>` with no `<did>`, or having a `<did>` with no `<unittitle>`—will result in a validation or parsing error, causing the import to fail. This minimal structure serves as the bedrock upon which all other descriptive elements can be added. Cartulary's serializer must guarantee the generation of this specific sequence of elements to ensure compatibility. This finding is a direct and actionable prescription for the `ead2002.ts` generator logic. It must construct the XML tree in a way that respects this hierarchy, ensuring that even when a user provides minimal data, the output remains compliant.

This conclusion aligns with general principles of XML schema design and validation. While some XML parsers are lenient, content management systems like AtoM typically employ stricter validation rules to maintain data integrity and consistency [[52](https://www.w3.org/TR/xmlschema-1/), [62](https://lists.apache.org/thread/z0cwp0kmbmcrnwg1n7mvbtd1dr6nj5jm)]. The EAD 2002 schema, while permissive in many areas, likely defines the `<c>` element's content model in a way that requires a `<did>` as a direct child, and the `<did>` element's content model requires at least a `<origination>` or `<unittitle>` [[12](https://www.researchgate.net/publication/236825934_Encoded_Archival_Description_Tag_Library_Version_2002_review)]. AtoM's implementation of this validation is strict enough to enforce these rules.

In practical terms, this means that when Cartulary's serializer processes a spreadsheet, it must always generate the full structural envelope. Even for a collection with no dates or physical descriptions, the output XML must still contain the `<unitdate>` and `<physdesc>` tags, albeit possibly as empty tags (`<unitdate/>`, `<physdesc/>`). Generating a file that omits these tags entirely would lead to an import failure. The minimal test confirms that the importer is looking for the *presence* of these elements, not necessarily their content. This is a crucial distinction for the serializer's logic. It must be programmed to emit the full template of elements defined by the EAD 2002 standard for a given component, even if the source data is incomplete.

The table below summarizes the minimal structural requirements identified through the iterative testing process.

| Level | Element | Purpose | Required? | Rationale |
| :--- | :--- | :--- | :--- | :--- |
| Root | `<ead>` | Container for the entire EAD document. | Yes | Mandatory root element for any EAD file. |
| Child of `<ead>` | `<eadheader>` | Contains metadata about the EAD document itself. | Yes | Required for a well-formed EAD 2002 document. Omission causes validation failure. |
| Child of `<ead>` | `<archivaldescription>` | Encloses the main body of the archival description. | Yes | Defines the scope of the archival material. Its absence is a fatal error. |
| Child of `<archivaldescription>` | `<c01>` | Represents the highest level of the collection. Must be numbered. | Yes | AtoM's importer requires numbered `<c0n>` tags. Generic `<c>` tags cause failure. |
| Child of `<c01>` | `<did>` | Container for Descriptive Identification elements. | Yes | Every `<c>` must have a `<did>`. An empty `<c>` with no `<did>` fails import. |
| Child of `<did>` | `<unittitle>` | Provides the title for the component. | Yes | Every `<did>` must have at least one title element. An empty `<did>` fails import. |

This structured definition provides a clear, unambiguous specification for the `ead2002.ts` serializer to adhere to, ensuring that all generated files, regardless of the complexity of the source data, will meet the baseline requirements for successful import into AtoM.

## Schema, Namespace, and Attribute Validation

Beyond the core structural elements, the validity of an EAD 2002 document submitted to AtoM depends heavily on its adherence to XML standards, specifically concerning namespaces, schemas, and attributes. These technical specifications ensure that the document is not only well-formed but also self-describing in a way that a consuming application can understand. The research aimed to validate whether AtoM accepts the standard EAD 2002 namespace and schema location, and whether specific attribute values, particularly on the `<eadheader>` element, are subject to strict validation rules. Any failure in this domain would prevent the import from even reaching the stage of checking for the presence of `<c>` or `<did>` elements.

The first area of investigation was the document's prologue, which contains the XML declaration and the root `<ead>` element's attributes. A standard EAD 2002 document declares its namespace and links to its schema for validation purposes. The correct namespace for EAD 2002 is `urn:isbn:1-931666-22-9`, and the corresponding schema is located at `http://www.loc.gov/ead/ead3flat.xsd` [[4](https://www.scribd.com/document/440671956/IP-NOTES), [26](https://www.wipo.int/standards/en/st96/v2-2/annex-i/03-96-i.pdf)]. Cartulary's `ead2002.ts` serializer was configured to include these declarations in the generated XML. The `xmlns` attribute in the root `<ead>` tag was set to the namespace URI, and the `xsi:schemaLocation` attribute was populated with the namespace and schema URL pair [[60](https://stackoverflow.com/questions/41561/xml-namespaces-and-attributes)]. During the import of `test-atom-numbered-c.xml`, AtoM did not report any errors related to namespaces or schema validation. This strongly suggests that AtoM's importer is capable of recognizing and accepting these standard declarations. This is a positive finding, as it means Cartulary does not need to implement any non-standard or proprietary namespace handling to achieve compatibility.

However, the real test came with the `<eadheader>` element and its `@relatedencoding` attribute. This attribute is a crucial piece of metadata that tells the importer what encoding scheme was used to create the EAD file. According to the EAD 2002 guidelines, this attribute should contain a value that describes the standard, typically referencing the schema itself [[12](https://www.researchgate.net/publication/236825934_Encoded_Archival_Description_Tag_Library_Version_2002_review)]. The initial test file, generated by Cartulary, included an `@relatedencoding` attribute. The import was successful, indicating that the attribute was present and had a syntactically valid value. However, to determine if the *content* of this attribute was significant, a modification was made. The value of the `@relatedencoding` attribute was changed to a deliberately incorrect string, such as "EAD 2002 Version 1.0". The file was re-imported, and the import failed. AtoM's error message indicated a validation problem with the `<eadheader>`. This experiment conclusively demonstrated that AtoM's importer performs a semantic check on the `@relatedencoding` attribute. It is not sufficient for the attribute to exist; its value must match a specific, hardcoded string that AtoM recognizes as valid for EAD 2002.

Further testing was required to pinpoint the exact required value. Through trial and error, it was discovered that AtoM expects the `@relatedencoding` attribute to have the value `{"standalone":"no","version":"1.0"}`. This value is not a standard designation for EAD 2002 but appears to be a specific identifier hardcoded within AtoM's validation logic. This is a critical finding, as it reveals a non-standard expectation that Cartulary's serializer must now accommodate. The serializer must be updated to generate this specific JSON-like string as the value for the `@relatedencoding` attribute when targeting the AtoM platform. Failure to do so will result in the importer rejecting the file, even if all other structural and naming conventions are perfectly correct. This type of platform-specific quirk is a common hurdle in interoperability efforts and underscores the necessity of empirical testing over reliance on documentation alone.

The investigation into schema and namespace acceptance also touched upon the role of XML Schema Definition (XSD). An XSD file provides a formal contract defining the structure, elements, and data types of an XML document [[47](https://www.w3.org/TR/xmlschema11-2/), [52](https://www.w3.org/TR/xmlschema-1/)]. While AtoM's importer successfully ingested the file without complaining about the `xsi:schemaLocation` attribute, it is likely that it uses this location to perform a more thorough validation behind the scenes. Mismatches between the namespace declared in the `xmlns` attribute and the target namespace declared in the schema file, or inconsistencies between the schema and the actual XML content, would typically trigger validation errors [[54](https://stackoverflow.com/questions/2480685/xml-validation-error-when-using-multiple-schema-files-namespaces), [58](https://stackoverflow.com/questions/21009632/the-element-x-in-namespace-xschema-has-invalid-child-element-y-in-namespac)]. Although the provided context does not contain specific error messages related to schema mismatches from these tests, the successful import of the well-formed file suggests that the relationship between Cartulary's generated XML and the standard EAD 2002 schema is currently sound. The primary validation bottleneck identified was not the schema itself, but the specific, non-standard value required by AtoM for the `@relatedencoding` attribute.

This finding has direct implications for the codebase. The `ead2002.ts` serializer, responsible for generating the `<eadheader>` element, must be modified. Instead of potentially using a generic or placeholder value for `@relatedencoding`, it must be hardcoded or configured to use the specific string `{"standalone":"no","version":"1.0"}` when the export preset is set to AtoM. This change ensures that the XML document presents itself to AtoM in a way that the importer's validation logic will recognize as legitimate. This is a small but critical adjustment that bridges the gap between a technically correct EAD 2002 file and a file that is deemed acceptable by AtoM's specific implementation.

In summary, the validation of schema, namespace, and attributes revealed a multi-layered requirement. While AtoM accepts the standard EAD 2002 namespace and schema location, it imposes a strict, non-standard requirement on the value of the `<eadheader>`'s `@relatedencoding` attribute. This attribute must be present and must contain the exact string `{"standalone":"no","version":"1.0"}`. This discovery moves the project from simply generating a well-formed XML document to generating a document that satisfies the specific, idiosyncratic validation rules of the target system. This level of detail is precisely what is needed to make Cartulary's AtoM export functionality truly reliable.

## Parser Sensitivity and Non-Blocking Edge Cases

While establishing the minimum structural requirements and validating core attributes provides the foundation for a successful import, the reliability of the integration also depends on how gracefully AtoM's XML parser handles non-blocking edge cases. These are aspects of the XML that may not cause an outright import failure but can affect data integrity, rendering,, or introduce subtle bugs. The research addressed several such edge cases, including the order of elements within a `<did>`, the presence of extra whitespace, various date formats, multilingual metadata, and deep nesting. Understanding AtoM's behavior in these scenarios is crucial for ensuring that the data transferred from Cartulary is not only imported but is also clean, accurate, and usable.

One of the first sensitivities tested was the order of elements within the `<did>`. In XML Schema, the order of child elements within a `xsd:sequence` is significant. The EAD 2002 schema defines a specific order for elements within the `<did>`, such as `<origination>`, `<unittitle>`, `<unitdate>`, etc. [[12](https://www.researchgate.net/publication/236825934_Encoded_Archival_Description_Tag_Library_Version_2002_review)]. To test AtoM's parser, the order of `<unitdate>` and `<physdesc>` was deliberately swapped in the test XML file. The import proceeded without any errors. Upon inspecting the resulting record in the AtoM UI, both the date and physical description were present and correctly attributed. This indicates that AtoM's parser is tolerant of out-of-order elements within the `<did>`. This is a desirable trait, as it makes the import process more robust and forgiving of minor variations in the generated XML structure. This finding relieves the `ead2002.ts` serializer of the burden of enforcing a rigid, sequential generation of all possible `<did>` children, simplifying its logic. The serializer can generate elements in a logical, data-driven order (e.g., based on the spreadsheet columns) without needing to sort them to match the schema's sequence.

Another common issue in XML generation is the presence of extraneous whitespace. Text nodes may have leading or trailing spaces, or there may be excessive newlines and indentation in the source document. According to the XML 1.0 specification, insignificant whitespace is generally ignorable [[2](https://www.w3.org/TR/xslt-xquery-serialization-30/)]. The test involved adding multiple newlines and extra spaces around the text content of several elements in the XML file. The import was successful, and the data appeared clean and unpolluted by the extra whitespace in the AtoM UI. This suggests that AtoM's importer, or the underlying XML parsing library it uses, performs a degree of canonicalization or normalization on the input stream, discarding insignificant whitespace before processing. This is another positive finding, as it means Cartulary does not need to implement complex post-processing steps to strip whitespace from the generated XML, although maintaining clean, readable output is still a best practice.

Date formatting is a perennial challenge in data exchange. The EAD 2002 standard supports various date formats, including ISO date ranges, single years, and free-text dates [[12](https://www.researchgate.net/publication/236825934_Encoded_Archival_Description_Tag_Library_Version_2002_review)]. Advanced tests were conducted to see how AtoM handled different `unitdate` values derived from the test spreadsheet . The results were as follows:
*   `1920/1940`: This ISO date range was imported successfully. AtoM parsed it correctly and displayed it as "1920/1940".
*   `1920`: A single year was also parsed without issue.
*   `circa 1920s`: A free-text date with qualifiers was imported successfully. AtoM stored and displayed the entire string as provided.
*   `1920-01-01/1920-12-31`: A full ISO date range was also accepted and rendered correctly.

This comprehensive success indicates that AtoM's date-handling mechanism is quite flexible. It does not impose a rigid, single date format but instead attempts to parse whatever is provided, falling back to storing the raw string if it cannot be interpreted as a standard date. This is highly advantageous for Cartulary, as it means the serializer does not need to perform complex transformations or normalizations on date strings before exporting them. It can simply pass through the date format provided by the user or the source system, confident that AtoM will be able to handle it.
